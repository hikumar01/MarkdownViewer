## 1. Diagram Zoom and Pan

Rendered Mermaid diagrams can be zoomed and panned interactively. This addresses a key gap in GitHub's and VS Code's preview.

**Acceptance Criteria**

- Mouse scroll wheel (or trackpad pinch) zooms in/out on the diagram under the cursor
- Click-and-drag pans the diagram when zoomed in beyond its natural size
- Double-click resets to the original fit-to-container size
- Minimum zoom: 50% of natural size. Maximum zoom: 400%
- Zoom and pan state resets when the document reloads
- Diagrams at their natural size are not draggable (no accidental panning)
- Zoom behavior does not affect the document scroll — the scroll wheel only zooms when the cursor is over the diagram
- Works for all diagram types

**Technical Notes**

Apply zoom/pan to the `<svg>` element inside each diagram container using a pointer event handler. A lightweight library such as `svg-pan-zoom` or a custom implementation using `transform: scale()` and `translate()` on a wrapper `<div>` is suitable.

```js
function initDiagramInteraction(container) {
  let scale = 1, originX = 0, originY = 0, isDragging = false

  container.addEventListener('wheel', (e) => {
    e.preventDefault() // prevent document scroll
    scale = Math.min(4, Math.max(0.5, scale - e.deltaY * 0.001))
    applyTransform()
  }, { passive: false })

  container.addEventListener('mousedown', startDrag)
  container.addEventListener('dblclick', () => { scale = 1; originX = 0; originY = 0; applyTransform() })
}
```

Wrap each diagram SVG in a `<div class="diagram-viewport">` with `overflow: hidden` to clip panned content.

**Dependencies:** Mermaid rendering (P0 Feature 2)

## 2. Copy Diagram as SVG / PNG

Right-clicking a rendered Mermaid diagram shows a context menu with copy options.

**Acceptance Criteria**

- Right-clicking a diagram shows a context menu with:
  - "Copy as SVG" — copies the SVG source to the clipboard
  - "Copy as PNG" — copies a rasterized PNG image to the clipboard
  - "Save as SVG…" — opens a save dialog to write the SVG file to disk
  - "Save as PNG…" — opens a save dialog to write the PNG file to disk
- Copied SVG is valid, self-contained (styles are inlined), and renders correctly when pasted into a vector editor
- Copied PNG is at 2× pixel density (retina resolution)
- Context menu appears only when right-clicking a diagram — not on prose or code blocks
- The default browser context menu is suppressed within diagram containers

**Technical Notes**

To rasterize SVG to PNG, draw the SVG onto a `<canvas>` using an `Image` element:

```js
async function svgToPng(svgElement, scale = 2) {
  const { width, height } = svgElement.getBoundingClientRect()
  const canvas = document.createElement('canvas')
  canvas.width = width * scale; canvas.height = height * scale
  const ctx = canvas.getContext('2d')
  const blob = new Blob([svgElement.outerHTML], { type: 'image/svg+xml' })
  const url = URL.createObjectURL(blob)
  await new Promise((resolve) => { const img = new Image(); img.onload = resolve; img.src = url })
  ctx.drawImage(img, 0, 0, canvas.width, canvas.height)
  URL.revokeObjectURL(url)
  return canvas
}
```

For clipboard write, use the Clipboard API with `ClipboardItem` for PNG. For SVG, write to clipboard as `text/plain` containing the SVG source.

For file save, use `@tauri-apps/plugin-dialog` in the renderer:

```typescript
import { save } from '@tauri-apps/plugin-dialog'
import { invoke } from '@tauri-apps/api/core'

const path = await save({ filters: [{ name: 'PNG Image', extensions: ['png'] }] })
if (path) await invoke('write_binary', { path, data: Array.from(pngBytes) })
```

**Dependencies:** Mermaid rendering (P0 Feature 2)

## 3. Click-to-expand Diagram (Lightbox)

Clicking a Mermaid diagram opens it full-screen for detailed inspection.

**Acceptance Criteria**

- Clicking a diagram opens a full-screen overlay (lightbox) containing the diagram at its full rendered size
- The lightbox background is a semi-transparent dark overlay
- The diagram inside the lightbox supports zoom and pan (P4 Feature 1)
- Pressing `Escape` or clicking outside the diagram closes the lightbox
- The diagram title (from a preceding heading or a caption) is optionally shown at the top of the lightbox
- Lightbox does not capture keyboard shortcuts used by the rest of the app (except `Escape`)
- Animation: overlay fades in over 150ms — respects `prefers-reduced-motion` (instant if set)

**Technical Notes**

The lightbox is a `<div class="lightbox-overlay">` inserted into the document `<body>` on click and removed on close. It is not a modal dialog (no `role="dialog"` required for this use case).

```js
diagramContainer.addEventListener('click', () => {
  const overlay = document.createElement('div')
  overlay.className = 'lightbox-overlay'
  overlay.innerHTML = `<div class="lightbox-content">${svgClone.outerHTML}</div>`
  overlay.addEventListener('click', (e) => { if (e.target === overlay) overlay.remove() })
  document.addEventListener('keydown', (e) => { if (e.key === 'Escape') overlay.remove() }, { once: true })
  document.body.appendChild(overlay)
  initDiagramInteraction(overlay.querySelector('.lightbox-content'))
})
```

**Dependencies:** Diagram zoom and pan (Feature 1 in this file), Mermaid rendering (P0 Feature 2)

## 4. Back / Forward Navigation History

After navigating via anchor links, the user can go back and forward through visited positions.

**Acceptance Criteria**

- After clicking an anchor link (`[link](#section)`), `Cmd+[` (or the back button if toolbar exists) returns to the previous scroll position
- `Cmd+]` moves forward through the history
- History depth: last **20** positions
- History is per-session — cleared when a new file is opened or the app restarts
- History entries store scroll position, not heading ID (scroll position is more reliable for documents that change)
- History does not track every scroll event — only anchor-link navigations

**Technical Notes**

Maintain a simple position history stack in the renderer:

```js
const history = { stack: [], index: -1 }

function pushHistory(scrollY) {
  history.stack = history.stack.slice(0, history.index + 1)
  history.stack.push(scrollY)
  history.index = history.stack.length - 1
  if (history.stack.length > 20) { history.stack.shift(); history.index-- }
}

function goBack() {
  if (history.index > 0) { history.index--; window.scrollTo({ top: history.stack[history.index] }) }
}
```

Call `pushHistory(window.scrollY)` immediately before each anchor-link navigation.

**Dependencies:** Anchor links (P1 Feature 5)

## 5. Incremental Re-render

On file change, only the blocks that changed are re-rendered — not the full document. This avoids a full-page flash for small edits.

**Acceptance Criteria**

- After a file change, prose sections that are identical to the previous render remain in the DOM untouched
- Only changed blocks are replaced in place
- Scroll position is preserved (no jump) during the incremental update
- Mermaid diagrams are only re-rendered if their source definition changed
- Code blocks are only re-highlighted if their content changed
- If the structural change is too large (e.g., a heading is added that shifts all block IDs), full re-render is used as a fallback — no corruption

**Technical Notes**

Assign stable IDs to each top-level block based on its content hash (not position). After each render, diff the new block list against the previous:

```js
function diffAndPatch(oldBlocks, newBlocks) {
  newBlocks.forEach((newBlock, i) => {
    const oldBlock = oldBlocks.find(b => b.id === newBlock.id)
    if (!oldBlock) {
      insertBlock(newBlock, i) // new block
    } else if (oldBlock.hash !== newBlock.hash) {
      replaceBlock(oldBlock.element, newBlock) // changed block
    }
    // unchanged blocks: no DOM touch
  })
  oldBlocks.filter(b => !newBlocks.find(nb => nb.id === b.id)).forEach(removeBlock)
}
```

This approach is similar to virtual DOM diffing but applied at the markdown block level. The block boundary is the top-level AST node (paragraph, heading, code block, blockquote, list, etc.).

**Dependencies:** Auto-reload (P1 Feature 3), debounce (P2 Feature 4), scroll preservation (P2 Feature 5)

## 6. Large File Handling

Files over 1 MB render without blocking the UI thread or causing the app to become unresponsive.

**Acceptance Criteria**

- Files up to **5 MB** render completely (no truncation)
- The UI remains responsive during rendering — no spinning beach ball or frozen window
- A loading indicator is shown for renders that take longer than **500ms**
- Mermaid diagrams in large files are rendered progressively (diagrams near the viewport first)
- Syntax highlighting in large files with many code blocks does not block the main thread
- Files over 5 MB show a warning: "Large file — some features may be limited" and render with Mermaid and syntax highlighting disabled, prose only

**Technical Notes**

Move the markdown-to-HTML rendering pipeline to a Web Worker (off the renderer main thread). In Tauri the rendering pipeline runs entirely in the WebView — the Rust backend is not involved in HTML generation, so a Web Worker is the right off-thread option:

```js
// renderer
const worker = new Worker('render-worker.js')
worker.postMessage({ markdown: fileContent, basedir: fileDir })
worker.onmessage = ({ data: html }) => injectHtml(html)
```

For progressive Mermaid rendering, use an `IntersectionObserver` to render diagrams only when they scroll near the viewport:

```js
const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) renderDiagram(entry.target)
  })
}, { rootMargin: '200px' })

document.querySelectorAll('.mermaid-source').forEach(el => observer.observe(el))
```

**Dependencies:** remark pipeline (P0 Feature 1), Mermaid rendering (P0 Feature 2)

## 7. Emoji Shortcodes

`:emoji_name:` shortcodes in markdown render as their corresponding Unicode emoji characters.

**Acceptance Criteria**

- Standard GitHub emoji shortcodes render: `:rocket:` → 🚀, `:tada:` → 🎉, `:warning:` → ⚠️
- Unrecognized shortcodes (`:notanemoji:`) are left as plain text — no error
- Emoji render inline within paragraphs, headings, list items, table cells
- Emoji do not render inside fenced code blocks or inline code spans
- The complete GitHub emoji set is supported (approximately 1900 entries)

**Technical Notes**

Use `remark-emoji` which maps GitHub's shortcode list to Unicode code points:

```js
import remarkEmoji from 'remark-emoji'
processor.use(remarkEmoji, { accessible: true })
```

The `accessible: true` option wraps each emoji in `<span role="img" aria-label="rocket">` for screen reader compatibility.

**Dependencies:** remark pipeline (P0 Feature 1)

## Open Points

### OP-16 — Diagram SVG Accessibility
Rendered Mermaid SVGs have no text alternative for screen readers. The following must be added to every diagram container:

- `role="img"` on the `<svg>` element or its wrapper `<div>`
- A `<title>` element as the first child of `<svg>` containing the diagram type (e.g., `"Flowchart diagram"`, `"Sequence diagram"`)
- If a heading immediately precedes the diagram in the document, use that heading text as an `aria-describedby` target for additional context
- If the diagram has a Mermaid `title` directive (e.g., `flowchart TD\n    title My Diagram`), use that as the `<title>` content

This is consistent with the `accessible: true` pattern used in Feature 7 (emoji) and should be implemented in the same post-render pass that injects diagram SVG into the DOM.

### OP-17 — Error State and Loading State Design Spec
Error and loading states are defined feature-by-feature with inconsistent UI patterns across the requirements. A unified component spec is needed before implementation begins:

**Error patterns referenced across requirements:**

| Pattern | Used in | Description |
|---|---|---|
| **Toast** | P1 F5, P1 F7, P3 F7, P5 F7 | Non-blocking notification; appears top-center; auto-dismisses after 4s; manually dismissible; max 3 stacked |
| **Inline error block** | P0 F3, P5 F7 | Rendered in place of failed content; red/warning border + icon + message + raw source in code block |
| **Placeholder** | P0 F8, P1 F4 | Neutral grey box with icon and short label; used for missing images and unknown diagram types |

**Loading patterns referenced across requirements:**

| Pattern | Used in | Description |
|---|---|---|
| **Spinner / skeleton** | P0 F2, P4 F6 | Shown while content is rendering; appears after 500ms delay to avoid flash for fast renders |
| **Progress indicator** | P4 F6 | For large files; linear progress bar at the top of the content area |

All six patterns should be defined as shared UI components before any feature implementation begins to ensure visual consistency.
