## 1. GitHub-style Callouts / Alerts

Blockquotes using GitHub's alert syntax render as styled callout boxes rather than plain blockquotes.

**Syntax**

```markdown
> [!NOTE]
> This is informational content.

> [!WARNING]
> This action cannot be undone.
```

**Acceptance Criteria**

- Five types supported: `NOTE`, `TIP`, `IMPORTANT`, `WARNING`, `CAUTION`
- Each type renders with:
  - A distinct accent color (left border + icon background)
  - An inline SVG icon matching GitHub's icon set
  - The type label as a bold heading within the callout
- Type-to-color mapping:
  - `NOTE` → blue
  - `TIP` → green
  - `IMPORTANT` → purple
  - `WARNING` → yellow/amber
  - `CAUTION` → red/orange
- Light and dark mode color variants for all five types
- Multi-paragraph content inside a callout renders correctly
- Nested markdown inside a callout renders: lists, code spans, bold, inline links
- An unknown `[!TYPE]` (not one of the five above) falls back to a plain blockquote — no error
- The type keyword is case-insensitive: `[!note]` = `[!NOTE]`

**Technical Notes**

Implement as a remark plugin that transforms blockquote AST nodes:

```js
function remarkGithubAlerts() {
  return (tree) => {
    visit(tree, 'blockquote', (node) => {
      const firstParagraph = node.children[0]
      const match = extractAlertType(firstParagraph) // returns 'NOTE' | 'TIP' | ... | null
      if (!match) return
      node.data = { hName: 'div', hProperties: { className: [`alert`, `alert-${match.toLowerCase()}`] } }
      // Remove the [!TYPE] line from rendered content
      firstParagraph.children = firstParagraph.children.slice(1)
    })
  }
}
```

Icons are inline SVGs embedded in CSS `::before` pseudo-elements or in the rehype output directly. Use GitHub's published icon paths for visual consistency.

**Dependencies:** remark pipeline (P0 Feature 1)

## 2. Image Captions

Images with a title string render as `<figure>` / `<figcaption>` instead of a bare `<img>`.

**Syntax**

```markdown
![alt text](./diagram.png "This is the caption")
```

**Acceptance Criteria**

- `![alt](src "caption")` where the image is a **standalone block** (sole element in a paragraph) renders as:
  ```html
  <figure>
    <img src="..." alt="alt text">
    <figcaption>This is the caption</figcaption>
  </figure>
  ```
- Inline images (image embedded within a sentence alongside other text) are **not** wrapped in `<figure>` — they render as a plain `<img>` with a `title` attribute
- Caption text supports basic inline markdown: bold, italic, inline code, links
- Caption is styled: centered, font-size 90% of body, muted color, top margin 4px
- Images without a title string render as plain `<img>` — no figcaption
- Empty title string `![alt](src "")` renders as plain `<img>`

**Technical Notes**

Implement as a rehype plugin that visits `<p>` nodes containing a single `<img>` child with a non-empty `title` property:

```js
function rehypeFigure() {
  return (tree) => {
    visit(tree, 'element', (node, index, parent) => {
      if (node.tagName !== 'p') return
      const [child] = node.children.filter(n => n.type !== 'text' || n.value.trim())
      if (!child || child.tagName !== 'img' || !child.properties.title) return

      // Replace <p><img title="..."></p> with <figure><img><figcaption>...</figcaption></figure>
      node.tagName = 'figure'
      node.children = [
        child,
        { type: 'element', tagName: 'figcaption', properties: {},
          children: [{ type: 'text', value: child.properties.title }] }
      ]
      delete child.properties.title
    })
  }
}
```

**Dependencies:** remark pipeline (P0 Feature 1), local image rendering (P0 Feature 8)

## 3. Image Sizing — Pandoc Syntax

Authors can specify image dimensions using Pandoc's attribute syntax.

**Syntax**

```markdown
![diagram](./arch.png){width=600}
![logo](./logo.png){width=50%}
![chart](./data.png){width=400 height=300}
```

**Acceptance Criteria**

- `{width=N}` sets the image width in pixels; height scales proportionally
- `{width=N%}` sets the image width as a percentage of its container
- `{width=N height=M}` constrains both dimensions; image is not stretched (`object-fit: contain`)
- Attributes are applied as inline `style` on the `<img>` element
- Malformed attribute block `{width=}` or `{not-an-attr}` is ignored — image renders at natural size
- Attribute block does not appear as visible text in the rendered output
- Standard image syntax without `{}` is unaffected
- Obsidian pipe syntax (`![alt|400](url)`) is **not** supported — Pandoc only

**Technical Notes**

Implement as a remark plugin that post-processes image nodes. The `{}` block immediately follows the closing `)` of the image syntax with no space:

```
![alt](url){width=600}
       ^^^^^^^^^^^^ parsed as image title by some parsers — needs custom handling
```

Preferred approach: process in the rehype layer. Use a regex on the raw markdown to extract `{...}` blocks adjacent to image syntax before the remark pipeline runs, attaching the parsed attributes to the image node via `data.hProperties`.

Alternatively, use `remark-directive` if attribute syntax overlaps; however, a targeted regex on image nodes is simpler and less likely to conflict with other plugins.

Width/height values are set as `style` attributes, not HTML `width`/`height` attributes, for CSS compatibility:

```js
node.properties.style = `width: ${value}; height: auto;`
```

**Dependencies:** remark pipeline (P0 Feature 1), image captions (Feature 2 in this file — both modify `<img>` nodes; run sizing before captions)

## 4. Debounced Re-render

Rapid file saves (e.g., auto-save while typing) do not cause visible flickering — re-renders are batched.

**Acceptance Criteria**

- Re-render is triggered no more than once per **300ms** window regardless of how many file change events fire
- The **last** change in a burst always triggers a render (trailing-edge debounce)
- Visual flickering or blank intermediate states are not visible to the user during rapid saves
- A loading indicator is optional — if shown, it appears only after 500ms of waiting (to avoid flashing for fast renders)
- The debounce timer resets on each new file-change event

**Technical Notes**

Debounce in the Rust backend before emitting the Tauri event to the renderer:

```rust
// Debounce using tokio::time::sleep — coalesce rapid successive change events
let (tx, mut rx) = tokio::sync::mpsc::channel::<()>(1);
tokio::spawn(async move {
    while rx.recv().await.is_some() {
        // drain any queued events within the 300ms window
        while let Ok(_) = rx.try_recv() {}
        tokio::time::sleep(std::time::Duration::from_millis(300)).await;
        window.emit("file-changed", &path_clone).unwrap();
    }
});
// In the notify watcher callback: tx.try_send(()).ok();
```

**Dependencies:** Auto-reload (P1 Feature 3)

## 5. Scroll Position Preservation on Reload

When the document re-renders due to a file change, the view stays at the same position rather than snapping to the top.

**Acceptance Criteria**

- After a reload triggered by a file change, the scroll position is within **one viewport height** of the previous position
- If the document becomes shorter than the previous scroll offset, the view scrolls to the bottom
- First load of a file always starts at the top
- Manual navigation (anchor clicks) is not affected — those scroll to the target heading
- Scroll position is not preserved across app restarts (that is a P3 enhancement of window state)

**Technical Notes**

Before injecting new HTML, capture the scroll offset. After injection, restore it:

```typescript
import { listen } from '@tauri-apps/api/event'
import { invoke } from '@tauri-apps/api/core'

await listen('file-changed', async () => {
  const scrollY = window.scrollY
  const newHtml = await invoke<string>('render_file')
  document.querySelector('.markdown-body')!.innerHTML = newHtml
  await runMermaidRender() // re-render diagrams in new content
  window.scrollTo({ top: scrollY, behavior: 'instant' })
})
```

The approach assumes the document structure is stable enough that the same pixel offset is meaningful. A heading-anchor approach (find the visible heading before reload and scroll to it after) is more robust for large documents but is a P3 enhancement.

**Dependencies:** Auto-reload (P1 Feature 3), debounced re-render (Feature 4 in this file)

## 6. Manual Theme Override

The user can pin the app to light or dark mode independently of the OS appearance setting.

**Acceptance Criteria**

- Three options available in the View menu: "Follow System" (default), "Light", "Dark"
- Selected option is persisted in user settings and restored on relaunch
- Switching the option takes effect immediately — no restart required
- When "Follow System" is selected, the app resumes responding to OS appearance changes (P1 Feature 1 behavior)
- When "Light" or "Dark" is pinned, OS appearance changes have no effect on the app
- Active option is indicated with a checkmark in the View menu

**Technical Notes**

Store the preference:

```js
// Values: 'system' | 'light' | 'dark'
store.set('themeOverride', 'system')
```

On launch and on menu selection:

```typescript
import { Store } from '@tauri-apps/plugin-store'
import { getCurrentWebviewWindow } from '@tauri-apps/api/webviewWindow'

const store = await Store.load('settings.json')
const appWindow = getCurrentWebviewWindow()

async function applyThemeOverride(override: 'system' | 'light' | 'dark') {
  await store.set('themeOverride', override)
  await store.save()
  if (override === 'system') {
    const systemTheme = await appWindow.theme()
    applyTheme(systemTheme ?? 'light')
  } else {
    applyTheme(override)
  }
}
```

The renderer applies the `dark` class to `<html>` via `applyTheme()` — same mechanism as P1 Feature 1. When "Follow System" is re-selected, the app re-subscribes to `onThemeChanged`.

**Dependencies:** Light/dark mode (P1 Feature 1)

## Open Points

### OP-12 — Callout Nesting
Feature 1 does not specify whether a callout (`> [!NOTE]`) can contain another callout. GitHub supports this. Decision needed:

- **Option A:** Supported — a callout inside another callout renders correctly, with the inner callout visually indented or inset.
- **Option B:** Not supported — an inner `[!TYPE]` inside an outer callout renders as a plain nested blockquote.

**Recommended: Option A** — the remark plugin that transforms blockquotes will naturally recurse into nested blockquotes if implemented correctly. No special handling is required; just ensure the CSS nesting styles are defined.

### OP-13 — Settings / Preferences Panel UI
At least 8 features reference "a setting in Preferences" or "persisted in settings" but no requirement specifies the Preferences UI itself. A dedicated spec is needed covering:

**Settings referenced across P2–P6:**

| Setting | Feature |
|---|---|
| Theme override (Light / Dark / System) | P2 Feature 6 |
| TOC visibility and depth | P3 Feature 1 |
| Status bar visibility | P3 Features 3–5 |
| Preferred editor path | P3 Feature 8 |
| Mermaid theme | P5 Feature 9 |
| Frontmatter panel collapse default | P5 Feature 8 |
| Font size | P6 Feature 3 |
| Custom CSS file path | P6 Feature 2 |

**UI decisions needed:**

- On macOS, the convention is `Cmd+,` opening a Preferences window (not an inline panel)
- Setting groups (suggested): **Appearance** (theme, font size, custom CSS) / **Viewer** (TOC depth, status bar, frontmatter) / **Editor Integration** (preferred editor) / **Diagrams** (Mermaid theme)
- All settings are global (not per-file) unless explicitly noted
- Preference changes take effect immediately without requiring a restart or document reload
