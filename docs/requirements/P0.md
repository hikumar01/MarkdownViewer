## 1. remark Rendering Pipeline

**Status: ✅ Complete**

The unified/remark/rehype pipeline is the core markdown processing engine. All other rendering features extend this pipeline.

**Implementation**

| File | What it does |
|------|-------------|
| `ui/renderer/pipeline.ts` | Defines and freezes the processor; exports `renderMarkdown(content, basePath)` |
| `ui/renderer/sanitize.ts` | Custom `sanitizeOptions` extending rehype-sanitize's defaultSchema |

The processor is built once at module load and frozen (`processor.freeze()`), which lets unified pre-compile the plugin chain so repeated `process()` calls are allocation-free.

**Actual pipeline order:**

```typescript
unified()
  .use(remarkParse)
  .use(remarkGfm)                                          // tables, strikethrough, task lists, autolinks
  .use(remarkRehype, { allowDangerousHtml: true })         // raw HTML nodes survive into rehype-raw
  .use(rehypeRaw)                                          // parse raw HTML nodes into proper HAST
  .use(rehypeExtractMermaid)                               // promote <pre><code.language-mermaid> to <pre.mermaid-source>
  .use(rehypeResolveImages)                                // rewrite relative img src to markdownviewer://
  .use(rehypeSanitize, sanitizeOptions)                    // remove dangerous HTML before Shiki runs
  .use(rehypeShiki, { themes: { light: 'github-light', dark: 'github-dark' } })
  .use(rehypeStringify)
  .freeze()
```

**Why this order matters:**

- `rehypeRaw` must come before `rehypeExtractMermaid` so that mermaid blocks written as raw HTML in markdown are also caught.
- `rehypeExtractMermaid` must come before `rehypeSanitize` so that `pre.mermaid-source` (which needs a non-standard `className` property) is emitted before sanitization strips it.
- `rehypeSanitize` must come before `rehypeShiki` so Shiki never processes unsanitized tokens. Shiki's `style=` output on `<span>` and `<pre>` is explicitly allowed in `sanitizeOptions` for those two elements.
- `basePath` (the directory of the open file) is passed per-call via `VFile.data` rather than a plugin option, because the processor is frozen and shared across all renders.

**Note on `rehypeExtractMermaid`:** The spec originally described using MDAST `hName`/`hChildren` data properties on remark nodes. That approach was abandoned because `mdast-util-to-hast`'s `applyData` wraps (rather than replaces) the default `<pre>` node, producing `<pre><pre><code>>` double-nesting. The current HAST-level plugin runs after `remarkRehype`, finds the `<pre><code class="language-mermaid">` that `remarkRehype` already produced, and promotes the class to the `<pre>` element. This avoids the nesting issue entirely.

**`sanitizeOptions` (`ui/renderer/sanitize.ts`)** extends `defaultSchema`:
- `class` allowed on all elements (GFM task lists, heading anchors)
- `style` allowed on `<span>` and `<pre>` only (Shiki inline colors)
- `id` allowed on `<h1>`–`<h6>` only (in-page anchor navigation)
- `className` allowed on `<pre>` (needed for `mermaid-source` class to survive sanitization)

**Acceptance Criteria**

- ✅ CommonMark spec compliant
- ✅ GFM extensions enabled: tables, strikethrough (`~~`), task lists (`[ ]`/`[x]`), autolinks
- ✅ Tables render with column alignment
- ✅ Nested markdown works inside table cells
- ✅ Blockquotes, ordered/unordered lists with correct nesting
- ✅ Inline elements: bold, italic, inline code, links, images, hard line breaks
- ✅ Setext and ATX headings
- ✅ Horizontal rules
- ✅ Unrecognized syntax passed through as plain text

## 2. Mermaid Diagram Rendering

**Status: ✅ Complete**

All Mermaid diagram types in ` ```mermaid ` fenced code blocks render as inline SVG.

**Implementation**

| File | What it does |
|------|-------------|
| `ui/renderer/mermaid.ts` | `initMermaid(theme)` and `renderMermaidBlocks(container)` |
| `ui/renderer/pipeline.ts` | `rehypeExtractMermaid` plugin marks mermaid blocks for deferred rendering |
| `ui/main.ts` | Calls `initMermaid` on load, `renderMermaidBlocks` after HTML is in the DOM |

**How the flow works:**

1. `rehypeExtractMermaid` (in the remark pipeline) finds `<pre><code class="language-mermaid">` and adds `class="mermaid-source"` to the `<pre>`. Shiki ignores this block because the class is no longer `language-mermaid`.
2. The pipeline returns HTML with `<pre class="mermaid-source"><code>diagram source</code></pre>` placeholders. A CSS loading animation (three blinking dots) is shown on these `<pre>` elements.
3. After `contentEl.innerHTML` is set, `renderMermaidBlocks(contentEl)` finds each `pre.mermaid-source`, calls `mermaid.render(id, source)`, sanitizes the returned SVG, and replaces the `<pre>` with a `<figure class="mermaid-diagram">`.
4. IDs are `mermaid-{counter}` using a module-level integer. Unique IDs prevent Mermaid from serving stale cached SVGs across re-renders.

**SVG sanitization — why not DOMPurify:**

Mermaid v11 renders node labels as `<foreignObject><div><span><p>` inside the SVG. DOMPurify's namespace validation removes HTML-namespace elements (div/span/p) that are children of SVG-namespace `foreignObject`, regardless of `ADD_TAGS` options, leaving every node box empty.

The actual sanitizer (`sanitizeSvgFragment` in `mermaid.ts`) works directly on DOM nodes:
1. Parse the SVG string via `div.innerHTML` (the HTML5 parser correctly switches to HTML content mode inside `<foreignObject>`)
2. Remove all `<script>` elements
3. Strip `on*` attributes and `javascript:` URL values
4. Move nodes into a `DocumentFragment` — never serialize back to a string

This avoids both DOMPurify's namespace-check stripping and WebKit's `innerHTML` serialization bug (WebKit drops HTML-namespace children of `foreignObject` when SVG is round-tripped through string serialization).

**Theme support:** `initMermaid` accepts `'default' | 'dark'`. When the OS theme changes, `main.ts` calls `initMermaid(newTheme)` then `reloadCurrentFile()` to re-render all diagrams.

**Acceptance Criteria**

- ✅ All listed diagram types (flowchart, sequence, class, state, ER, gantt, pie, git graph, user journey, mindmap, timeline, quadrant, XY chart)
- ✅ Inline SVG output (not `<img>` or `<canvas>`)
- ✅ Async — does not block prose rendering
- ✅ Diagram theme matches light/dark mode
- ✅ Re-renders on theme change
- ✅ Loading placeholder shown while diagram renders

## 3. Mermaid Parse Error Display

**Status: ⚠️ Partial** — See [unimplemented.md](unimplemented.md#mermaid-error-display) for the remaining work.

When a Mermaid definition has a syntax error, a broken-diagram placeholder appears inline.

**Implementation**

| File | What it does |
|------|-------------|
| `ui/renderer/mermaid.ts` | `catch` block in `renderMermaidBlocks` creates `figure.mermaid-broken` |
| `ui/styles/app.css` | `.mermaid-broken` — dashed border, broken-image icon, "Diagram error" label |

The `catch` block extracts the error message (Mermaid parse errors surface as `err.str`, not `err.message`) and sets it as the `title` attribute of the broken figure. The error is visible on hover as a tooltip and in the DevTools console.

**What's missing:** The error message is not shown as visible text inline, and the raw source block is not rendered. See [unimplemented.md](unimplemented.md#mermaid-error-display).

**Acceptance Criteria**

- ✅ Error block appears in place of the diagram
- ✅ Dashed border and broken-image icon
- ⚠️ Error message: visible in tooltip only, not inline text
- ❌ Raw source definition in a scrollable code block
- ✅ Other diagrams and content render normally
- ✅ Error clears when file is corrected and reloaded
- ❓ Empty blocks: not explicitly tested — may show error rather than placeholder

## 4. Syntax-Highlighted Code Blocks

**Status: ✅ Complete**

Fenced code blocks with a language identifier render with syntax highlighting using GitHub's token color palette.

**Implementation**

| File | What it does |
|------|-------------|
| `ui/renderer/pipeline.ts` | `rehypeShiki` with `{ themes: { light: 'github-light', dark: 'github-dark' } }` |
| `ui/renderer/sanitize.ts` | Allows `style` on `<span>` and `<pre>` for Shiki's inline color attributes |
| `ui/styles/app.css` | `html.dark .shiki, html.dark .shiki span` — activates dark palette |

**How dual-theme works:** Shiki emits CSS custom properties (`--shiki-light`, `--shiki-dark`, `--shiki-light-bg`, `--shiki-dark-bg`) on each `<span>`. The active theme is selected purely in CSS by toggling `html.dark` on the document element — no JavaScript re-render is needed when the theme changes. The `html.dark` class is set by `detectTheme()` in `ui/events/theme.ts`.

**Long lines:** `#content pre { overflow-x: auto }` in `app.css` ensures wide code blocks scroll horizontally rather than overflow the viewport.

**Acceptance Criteria**

- ✅ Languages: JS, TS, Python, Rust, Go, Java, C, C++, Bash, SQL, YAML, JSON, TOML, HTML, CSS, Markdown (Shiki bundles broad language support)
- ✅ Language tag is case-insensitive
- ✅ Unknown language tag renders as plain monospace
- ✅ No language tag renders as plain monospace
- ✅ Long lines scroll horizontally
- ✅ `github-light` / `github-dark` themes
- ✅ Theme switches on system theme change without reload
- ✅ ` ```mermaid ` blocks excluded (handled before Shiki runs)

## 5. Open File (Cmd+O)

**Status: ⚠️ Partial** — See [unimplemented.md](unimplemented.md#open-file-gaps) for the remaining work.

The native OS file open dialog lets the user select a `.md` or `.markdown` file.

**Implementation**

| File | What it does |
|------|-------------|
| `app/src/lib.rs` | Native menu handler calls `app.dialog().file()` and emits `open-file` event |
| `ui/main.ts` | Listens for `open-file` event and calls `loadFile(path)` |
| `app/src/commands.rs` | `read_file` command — reads and returns file content |

**How the dialog is triggered:** The "Open File" menu item and Cmd+O shortcut are wired up in the Rust menu builder (`lib.rs`). When activated, the Rust handler opens the native file picker, then emits `"open-file"` with the selected path. The frontend never calls a separate `open_file_dialog` command — the dialog is fully handled in Rust.

**What's missing:**
- The dialog does not start in the directory of the currently open file (`.set_directory()` not set)
- `loadFile` has no `try-catch` around `invoke('read_file', ...)` — a permissions error or race-deleted file will produce an uncaught promise rejection
- No `dialogMessage` shown to the user on read failure

See [unimplemented.md](unimplemented.md#open-file-gaps).

**Acceptance Criteria**

- ✅ Cmd+O opens the native file picker
- ✅ File filter: `.md` and `.markdown` — All Files also available
- ✅ "File → Open…" menu item triggers the dialog
- ✅ Selected file loads and renders immediately
- ✅ Dialog cancellation leaves the current file unchanged
- ❌ File read errors show a toast/dialog — currently crashes silently
- ❌ Dialog opens in the directory of the currently open file

## 6. Single Tab Window Model

**Status: ✅ Complete**

The app maintains a single window showing a single file.

**Implementation**

| File | What it does |
|------|-------------|
| `app/tauri.conf.json` | Defines a single `"main"` window; `minWidth: 600, minHeight: 400` enforced |
| `app/src/lib.rs` | `tauri_plugin_single_instance` — second launch passes args to running instance and exits |
| `app/src/lib.rs` | `WindowEvent::CloseRequested` → `api.prevent_close()` + `window.hide()` (macOS close-to-hide) |
| `ui/main.ts` | `listen('open-file')` replaces the current file in-place — no new window |

**Single-instance behavior:** If the user launches a second instance (e.g., double-clicking a file in Finder while the app is already running), `tauri_plugin_single_instance` calls the plugin callback with the new arguments, emits `"open-file"` to the existing window, and focuses that window. The second process exits.

**Close-to-hide (macOS):** Clicking the red window close button hides the window instead of destroying it. The Dock icon stays active. Clicking the Dock icon re-shows the window. This is standard macOS app behavior.

**Acceptance Criteria**

- ✅ Exactly one window at all times
- ✅ Opening any file replaces the current one in the same window
- ✅ Dock click (macOS) focuses the existing window
- ✅ Single-instance enforced
- ✅ Minimum window size: 600 × 400 px
- ✅ Window is resizable above the minimum

## 8. Local Image Rendering

**Status: ✅ Complete**

Images referenced with relative local paths resolve from the open file's directory on disk.

**Implementation**

| File | What it does |
|------|-------------|
| `app/src/protocol.rs` | `markdownviewer://` URI scheme handler — serves image bytes after security checks |
| `ui/renderer/pipeline.ts` | `rehypeResolveImages` plugin — rewrites relative `src` to `markdownviewer://` |
| `ui/main.ts` | `attachImageHandlers` — loading animation + broken-image placeholder |
| `ui/styles/app.css` | `.img-wrapper`, `.img-loading`, `.img-broken` styles |

**Security chain:**

1. `resolveLocalPath(basePath, src)` in `pipeline.ts` decodes percent-encoded characters first (to catch `%2e%2e` as `..`), rejects absolute paths and any path that resolves outside `basePath`. Returns `''` on rejection — the `<img>` keeps its original src and will fail to load.
2. The browser sends `markdownviewer://{resolved-path}` to the Rust protocol handler.
3. `protocol.rs` calls `canonicalize()` (rejects symlink traversal and missing files), checks `is_file()`, and validates the file extension against an allowlist (png, jpg, jpeg, gif, webp, svg, ico, bmp, tiff, avif). Returns 403 for anything else.

**Why `markdownviewer://` and not `file://`:** The `file://` scheme gives untrusted markdown content direct access to any local path. The custom scheme applies the canonicalization and extension guards before serving any bytes.

**Image loading states:**
- While loading: `div.img-wrapper.img-loading` wraps the `<img>` (which is invisible); a CSS dot-pulse animation is shown.
- Load success: `img-loading` class removed — image appears.
- Load failure: wrapper replaced with `div.img-broken` (dashed border, broken-image icon, "Image not found" label).

**Acceptance Criteria**

- ✅ `./relative` and `../parent` paths resolve from the file's directory
- ✅ Web URLs (`https://`, `http://`) rendered as-is
- ✅ Missing image: broken-image placeholder with path in tooltip
- ✅ Supported formats: PNG, JPG, GIF, WebP, SVG, AVIF (served as `<img>`, not inline SVG)
- ✅ No directory traversal outside the filesystem root
- ✅ Loading animation while image loads

## Open Points

### OP-01 — Security Model ✅ Resolved

Tauri v2's security model is secure by default. No manual `contextIsolation` configuration required.

- The WebView has no access to Node.js or Rust APIs by default
- All backend calls go through typed Tauri Commands (`invoke()`)
- The Capabilities file (`app/capabilities/default.json`) is an explicit allowlist — only listed commands can be called
- CSP enforced in `tauri.conf.json`

**Remaining action:** Before first release, scope `fs:read-files` to the user's home directory rather than the entire filesystem.

A welcome screen (`#welcome`) exists with "Open a file to get started" text and a hint about drag-and-drop or Cmd+O. No formal UX spec for what's disabled vs enabled with no file open. See [unimplemented.md](unimplemented.md#empty-state).

File and Edit menus are implemented. View and Window menus exist on macOS only. Help menu is missing entirely. See [unimplemented.md](unimplemented.md#menubar).

macOS close-to-hide is implemented. Before-quit pending-write handling is not applicable yet (no file writing exists). Crash recovery is not implemented. See [unimplemented.md](unimplemented.md#app-lifecycle).
