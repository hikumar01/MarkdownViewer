## 1. remark Rendering Pipeline

The unified/remark/rehype pipeline is the core markdown processing engine. All other rendering features extend this pipeline.

**Parser decision:** `remark` over `markdown-it`. Reason: micromark (remark's tokenizer) resolves `~` (subscript) vs `~~` (strikethrough) at the character-run level before any plugin runs, making plugin registration order irrelevant for delimiter conflicts.

**Acceptance Criteria**

- CommonMark spec compliant
- GFM extensions enabled: tables, strikethrough (`~~`), task lists (`[ ]`/`[x]`), autolinks
- Tables render with column alignment (left / center / right per column spec)
- Nested markdown works inside table cells (bold, inline code, links)
- Blockquotes render with correct nesting depth
- Ordered and unordered lists render with correct nesting
- Inline elements: bold (`**`), italic (`*`/`_`), inline code (`` ` ``), links, images, hard line breaks
- Setext and ATX headings both render correctly
- Horizontal rules render
- Unrecognized syntax is passed through as plain text — no crash

**Technical Notes**

Pipeline setup:

```js
import { unified } from 'unified'
import remarkParse from 'remark-parse'
import remarkGfm from 'remark-gfm'
import remarkRehype from 'remark-rehype'
import rehypeRaw from 'rehype-raw'
import rehypeSanitize from 'rehype-sanitize'
import rehypeStringify from 'rehype-stringify'

const processor = unified()
  .use(remarkParse)
  .use(remarkGfm)
  .use(remarkRehype, { allowDangerousHtml: true })  // pass raw HTML nodes to rehype-raw
  .use(rehypeRaw)          // parse raw HTML nodes into proper HAST elements
  .use(rehypeSanitize)     // sanitize — must come before Shiki
  .use(rehypeStringify)
```

Use remark v14+ (ESM). `remarkGfm` covers tables, strikethrough, task lists, and autolinks in one plugin. Additional syntax plugins slot in between `remarkGfm` and `remarkRehype`. `rehype-raw` + `rehype-sanitize` are required — without them, raw HTML in markdown either gets silently dropped (no `rehype-raw`) or rendered unsanitized (no `rehype-sanitize`).

**Dependencies:** None — this is the foundation for all other rendering features.

## 2. Mermaid Diagram Rendering

All Mermaid diagram types found in ```` ```mermaid ```` fenced code blocks are rendered as inline SVG.

**Acceptance Criteria**

- Supported diagram types:
  - Flowchart (`graph` / `flowchart TD`, `LR`, `BT`, `RL`)
  - Sequence diagram
  - Class diagram
  - State diagram (v1 and v2)
  - Entity Relationship diagram
  - Gantt chart
  - Pie chart
  - Git graph
  - User journey
  - Mindmap
  - Timeline
  - Quadrant chart
  - XY chart
- All diagrams render as inline SVG (not `<img>` or `<canvas>`)
- Rendering is async — does not block the UI thread or delay prose rendering
- Diagram theme matches the app's current light/dark mode
- When the app theme changes, all diagrams re-render with the updated theme
- A loading state (spinner or placeholder) is shown while the diagram renders

**Technical Notes**

Use Mermaid v11+. Initialize once on app load:

```js
mermaid.initialize({ startOnLoad: false, theme: 'default' })
```

Processing flow:
1. remark pipeline detects ```` ```mermaid ```` blocks and emits them as `<pre class="mermaid-source"><code>...</code></pre>` HAST elements (via `hName`/`hChildren` data properties, bypassing the raw-node path)
2. After HTML is injected into the DOM, a post-render pass calls `mermaid.render(id, definition)` for each `pre.mermaid-source` element
3. The returned SVG is sanitized with DOMPurify (SVG profile) and the `<pre>` is replaced by a `<figure class="mermaid-diagram">` containing the clean SVG
4. On theme change, re-initialize mermaid with the new theme and re-render by reloading the current file

Use an incrementing counter for diagram IDs (not a content hash) — Mermaid's internal cache is keyed by ID and does not deduplicate across re-renders, so unique IDs per render pass are simpler and more predictable.

**Dependencies:** remark pipeline (Feature 1)

## 3. Mermaid Parse Error Display

When a Mermaid definition has a syntax error, the error is displayed inline where the diagram would appear. The rest of the document renders normally.

**Acceptance Criteria**

- Error block appears in place of the diagram — document does not crash or stop rendering
- Error block shows:
  - A red/warning-colored border and icon
  - The Mermaid error message string
  - The raw source definition in a scrollable code block
- All other diagrams and document content render normally
- Error state clears when the file is corrected and reloaded
- Empty ```` ```mermaid ```` blocks (no content) show an "empty diagram" placeholder, not an error

**Technical Notes**

Wrap every `mermaid.render()` call:

```js
try {
  const { svg } = await mermaid.render(id, definition)
  const cleanSvg = DOMPurify.sanitize(svg, { USE_PROFILES: { svg: true, svgFilters: true } })
  const figure = document.createElement('figure')
  figure.innerHTML = cleanSvg
  pre.replaceWith(figure)
} catch (err) {
  const errorDiv = document.createElement('div')
  const msg = document.createElement('p')
  msg.textContent = err.message ?? err.str   // Mermaid parse errors surface as err.str
  errorDiv.appendChild(msg)
  pre.replaceWith(errorDiv)
}
```

`err.str` contains the human-readable Mermaid parse error. Do not expose the full stack trace. DOMPurify with the SVG profile blocks `<script>` and event-handler attributes that Mermaid's `securityLevel:'loose'` mode may produce.

**Dependencies:** Mermaid diagram rendering (Feature 2)

## 4. Syntax-Highlighted Code Blocks

Fenced code blocks with a language identifier render with syntax highlighting using GitHub's token color palette.

**Acceptance Criteria**

- Languages supported at minimum: JavaScript, TypeScript, Python, Rust, Go, Java, C, C++, Bash/Shell, SQL, YAML, JSON, TOML, HTML, CSS, Markdown
- Language tag is case-insensitive: ` ```JS ` and ` ```js ` both work
- Unknown language tag: renders as plain monospace — no error, no fallback language guessing
- No language tag: renders as plain monospace
- Long lines scroll horizontally — no word wrap
- Token colors use `github-light` theme in light mode and `github-dark` theme in dark mode
- Theme switches when app theme changes (no reload required)
- ```` ```mermaid ```` blocks are excluded — handled by Mermaid rendering (Feature 2)

**Technical Notes**

Use Shiki (preferred over `rehype-highlight` / highlight.js):
- Better token accuracy, same themes as VS Code
- Supports `github-light` and `github-dark` themes natively
- Bundle only the required language grammars to limit bundle size

```js
import rehypeShiki from '@shikijs/rehype'

processor.use(rehypeShiki, {
  themes: { light: 'github-light', dark: 'github-dark' }
})
```

Shiki applies `data-theme` attributes to code blocks; CSS switches the visible theme based on `@media (prefers-color-scheme: dark)` or the manual theme class.

**Dependencies:** remark pipeline (Feature 1)

## 5. Open File (Cmd+O)

The native OS file open dialog allows the user to select a `.md` or `.markdown` file.

**Acceptance Criteria**

- `Cmd+O` opens the native OS file picker dialog
- File filter defaults to `.md` and `.markdown` — "All Files" option also available
- "File → Open…" menu item triggers the same dialog
- Selected file loads and renders immediately
- Dialog cancellation leaves the current file unchanged
- File that cannot be read (permissions error, file deleted): show a toast notification with the error and the file path — do not crash
- The dialog opens in the directory of the currently open file (if one is open), otherwise in the user's home directory

**Technical Notes**

**Rust backend command:**

```rust
// app/src/commands.rs
#[tauri::command]
async fn open_file_dialog(app: AppHandle, current_path: Option<String>) -> Option<String> {
    let mut builder = FileDialogBuilder::new()
        .add_filter("Markdown", &["md", "markdown"])
        .add_filter("All Files", &["*"]);
    if let Some(p) = current_path {
        if let Some(dir) = Path::new(&p).parent() {
            builder = builder.set_directory(dir);
        }
    }
    builder.pick_file().map(|p| p.to_string_lossy().to_string())
}
```

**Frontend invocation:**

```typescript
import { invoke } from '@tauri-apps/api/core'
const path = await invoke<string | null>('open_file_dialog', {
  currentPath: currentFilePath
})
if (path) openFile(path)
```

File reading happens in Rust via a `read_file` command; the content string is returned to the frontend for rendering.

**Dependencies:** Window model (Feature 6)

## 6. Single Tab Window Model

The app maintains a single window showing a single file. Opening any file replaces the current one in the same window.

**Acceptance Criteria**

- Exactly one window exists at all times while the app is running
- Opening a file via File menu (Cmd+O), CLI argument, Finder, or file association replaces the current file in the same window — no new window is created
- Clicking the app icon in the Dock (macOS) while the app is running focuses the existing window; does not open a new window
- `tauri-plugin-single-instance` ensures only one process runs — a second launch passes its arguments to the running instance and exits
- Minimum window size enforced: **600 × 400 px**
- Window is resizable above the minimum

**Technical Notes**

```rust
// app/src/main.rs
tauri::Builder::default()
    .plugin(tauri_plugin_single_instance::init(|app, argv, _cwd| {
        // Second instance launched — extract file path from args and open it
        if let Some(path) = argv.get(1) {
            app.emit("open-file", path).unwrap();
        }
        // Focus the existing window
        app.get_webview_window("main").unwrap().set_focus().unwrap();
    }))
    .setup(|app| {
        let window = app.get_webview_window("main").unwrap();
        window.set_min_size(Some(PhysicalSize::new(600, 400)))?;
        Ok(())
    })
```

```json
// tauri.conf.json
{ "app": { "windows": [{ "minWidth": 600, "minHeight": 400 }] } }
```

macOS re-open (Dock click): Tauri handles `applicationShouldHandleReopen` automatically — it focuses the existing window without creating a new one.

**Dependencies:** None

## 8. Local Image Rendering

Images referenced with relative or absolute local paths resolve from the open file's directory on disk.

**Acceptance Criteria**

- `![](./assets/diagram.png)` resolves relative to the directory of the open `.md` file
- `![](../shared/logo.png)` resolves correctly using `..` traversal
- Absolute local paths resolve correctly on the file system
- Web URLs (`https://`, `http://`) are rendered as-is without interception
- Missing image: broken-image placeholder is shown with the attempted path displayed below it — never a silent blank space
- Supported formats: PNG, JPG/JPEG, GIF, WebP, SVG, AVIF
- SVG files render as `<img>` elements (not inline SVG) to prevent embedded script execution
- The custom protocol handler does not allow directory traversal outside the file system root

**Technical Notes**

Register the custom protocol in the Tauri Rust backend:

```rust
// app/src/protocol.rs
pub fn handle<R: tauri::Runtime>(
    _ctx: tauri::UriSchemeContext<'_, R>,
    request: Request<Vec<u8>>,
) -> Response<Vec<u8>> {
    let decoded = urlencoding::decode(request.uri().path())
        .map(|s| s.into_owned())
        .unwrap_or_default();
    let canonical = std::fs::canonicalize(&decoded)?;  // rejects traversal + missing files
    // Only serve known image extensions — never source code or secrets
    if !is_image_extension(&canonical) { return 403; }
    let mime = mime_guess::from_path(&canonical).first_or_octet_stream();
    Response::builder().header("Content-Type", mime).body(bytes).unwrap()
}
```

In the rehype pipeline (frontend TypeScript), rewrite relative image `src` attributes:

```typescript
// Resolve relative path against the open file's directory, reject traversal
const resolved = resolveLocalPath(basePath, src)  // returns '' on traversal
if (resolved) node.properties.src = `markdownviewer://${resolved}`
```

Configure CSP in `tauri.conf.json` to allow the `markdownviewer:` scheme:

```json
{
  "app": {
    "security": {
      "csp": "default-src 'self'; img-src 'self' markdownviewer: data: https:; style-src 'self' 'unsafe-inline'; script-src 'self'; frame-ancestors 'none'; base-uri 'self'"
    }
  }
}
```

**Dependencies:** remark pipeline (Feature 1), Window model (Feature 6)

## Open Points

### OP-01 — Security Model ✅ Resolved by Tauri v2

**Framework: Tauri v2.** The security model is secure by design — no manual `contextIsolation` or preload script configuration is required.

**How Tauri v2 security works:**

- The WebView has **no access to Node.js or Rust APIs by default** — the frontend is a sandboxed web page
- All backend access goes through **typed Tauri Commands** (`invoke()` in frontend → `#[tauri::command]` in Rust) — there is no untyped IPC channel
- The **Capabilities system** (`app/capabilities/default.json`) acts as an explicit allowlist — the frontend can only call commands that are listed; all others are rejected at the framework level
- CSP is configured in `tauri.conf.json` and enforced by the WebView process — not an afterthought
- No preload script needed — `invoke()` is injected by Tauri automatically with no Node.js bridge

**What the capabilities file must list for MarkdownViewer (see README.md plugin manifest):**

```json
{
  "identifier": "default",
  "windows": ["main"],
  "permissions": [
    "fs:read-files", "fs:write-files",
    "dialog:open", "dialog:save",
    "shell:open",
    "store:allow-get", "store:allow-set", "store:allow-save",
    "window-state:allow-restore-state",
    "single-instance:allow-init",
    "deep-link:allow-get-current"
  ]
}
```

**Remaining action:** Audit the capability list before first release to ensure no over-permissioned entries exist (e.g., `fs:read-files` should be scoped to the user's home directory, not the entire filesystem, using Tauri's scope configuration).

### OP-02 — Empty State / First-run Experience
No file specifies what a brand-new user sees before any file is opened. Needs a UX spec covering:

- What is shown in the content area with no file open (drop zone? "Open a file to get started" prompt?)
- Which menu items are disabled with no file open: `Cmd+E`, `Cmd+F`, `Cmd+Shift+T`, `Cmd+[`, `Cmd+]`, `Cmd+K` (heading results)
- Status bar behavior when no file is open (placeholder dashes or hidden)
- Where and how the first-launch "Set MarkdownViewer as default .md viewer?" prompt appears (modal, banner, or toast)
- Whether any onboarding walkthrough exists (recommended: no — keep the first run simple)

### OP-03 — App Menubar Structure
Menu items are referenced piecemeal across P0–P6 but no file provides a complete macOS menubar spec. A consolidated menu tree is needed before implementation, covering:

- Full hierarchy: **MarkdownViewer / File / Edit / View / Window / Help**
- Which items are disabled when no file is open
- All keyboard accelerators (see README.md for the consolidated shortcuts reference)
- Suggested top-level structure:
  - **File:** Open… / Open Recent ▶ / Open in Editor / — / Export ▶ (P7) / Close / Quit
  - **Edit:** Find… / Find Next / Find Previous / — / Copy as HTML
  - **View:** Toggle TOC / Toggle Status Bar / — / Zoom In / Zoom Out / Reset Zoom / — / Theme ▶ / Diagram Theme ▶ / — / Enter Presentation Mode (P7)
  - **Window:** Minimize / Zoom / — / Bring All to Front
  - **Help:** MarkdownViewer Help / Report Issue / — / About MarkdownViewer

### OP-04 — App Lifecycle
Partially covered in Feature 6 but incomplete:

- **macOS: app stays alive after window closes** — the process does not quit when the window is closed (standard macOS behavior). Dock click reopens the window. In Tauri, set `"closeRequestedHandler": true` and handle `CloseRequested` to hide rather than destroy the window. `RunEvent::ExitRequested` controls app-level quit.
- **Before-quit:** When the user quits (`Cmd+Q`), any in-flight file write (P5 Task Write-back) must complete before the process exits. Use `app.run(|_app, event| { if let RunEvent::ExitRequested { .. } = event { /* await pending writes */ } })` in Tauri's event loop.
- **Crash recovery:** If the app crashed in a previous session, does it attempt to reopen the last file? Recommended: yes, with a "MarkdownViewer quit unexpectedly — reopen last file?" prompt on next launch.

### OP-05 — Code Block Copy Button
Mentioned as "optional for P0 — see P3" in Feature 4 but never assigned to any priority file. Decision needed: assign a priority. **Recommended: P2** — a copy button on hover is expected in any modern code viewer and is low implementation cost with Shiki.

### OP-06 — Code Block Line Numbers
Not mentioned anywhere. Needs an explicit in-scope / out-of-scope decision. **Recommended: out of scope for v1** — line numbers add visual noise for prose-heavy docs. Revisit as a P6 setting (enable via Custom CSS or a toggle in Preferences).

### OP-07 — Wide Table Scroll Behavior
Feature 1 specifies tables render with proper column alignment but does not specify behavior when a table exceeds the 768px prose column width. Decision needed: **recommended — horizontal scroll** on a `<div class="table-wrapper">` wrapping element rather than clipping or reflowing the table.
