# P7 Requirements — Future Scope

These features are planned but not scheduled. Each requires a separate design and scoping exercise before implementation begins. They are documented here to inform architectural decisions made in earlier phases.

## 1. Export to PDF

The rendered document is exported as a PDF file.

**Rough scope**

- "File → Export as PDF…" opens a save dialog
- The PDF output matches the rendered view: prose, diagrams (SVG embedded), syntax-highlighted code
- Page margins, header/footer, and page size are configurable in an export settings panel
- Mermaid diagrams render as vector SVG in the PDF (not rasterized)
- Export uses Tauri's `WebviewWindow.print()` which triggers the OS print dialog (user selects "Save as PDF"); for a fully automated no-dialog path, a Chromium sidecar process via `tauri-plugin-shell` is the alternative

**Architectural note for earlier phases:** The renderer's HTML/CSS must be print-friendly. Add `@media print` styles even before this feature is built — it is low-cost and avoids a large refactor later.

---

## 2. Export to Self-contained HTML

The rendered document is exported as a single `.html` file with all styles and assets inlined.

**Rough scope**

- "File → Export as HTML…" opens a save dialog
- Output is a single `.html` file with no external dependencies
- CSS is inlined in a `<style>` block
- Local images are embedded as `data:` URIs (base64-encoded)
- Mermaid diagrams are embedded as inline SVG
- The output renders correctly in any modern browser without the markview app or its custom protocol

**Architectural note for earlier phases:** Keep the renderer's HTML output clean and standards-compliant. The custom `markview://` protocol should only be used for rendering — the export pipeline will need to resolve it back to absolute paths.

---

## 3. Export Individual Diagrams (PNG / SVG)

A diagram can be saved directly to a file from a context menu.

**Rough scope**

- Right-click on a diagram → "Save as PNG…" / "Save as SVG…"
- PNG export at 2× density (retina)
- SVG export is self-contained (no external font or style references)

**Note:** "Copy as SVG/PNG" is already in P4 (Feature 2). This feature adds the file-save equivalent. They share the same rasterization code — implement P4 Feature 2 generically so saving is a minor addition here.

## 4. Folder Sidebar and File Browser

A sidebar panel lists the files in a directory, allowing navigation between related documents.

**Rough scope**

- User opens a folder via "File → Open Folder…" (Cmd+Shift+O)
- Sidebar shows a file tree filtered to `.md` and `.markdown` files
- Clicking a file in the sidebar opens it in the viewer (replaces current file)
- Current file is highlighted in the sidebar
- Folder state (open/collapsed subdirectories, scroll position) persists per-folder in settings
- File watcher monitors the folder for new/deleted/renamed files and updates the sidebar

**Architectural note for earlier phases:** The single-file window model (P0 Feature 6) should not assume the sidebar will never exist. The content area should be designed with a collapsible left panel slot, even if it is empty in early phases.

---

## 5. Relative File Links

Clicking `[link text](./other.md)` in the rendered document opens the linked file in the viewer.

**Rough scope**

- Relative `.md` links resolve from the current file's directory
- Clicking a relative link loads the linked file in the same window (same single-tab model)
- Back/forward navigation history (P4 Feature 4) records the file change as a navigable step
- Anchor links in relative paths (`./other.md#section`) scroll to the heading after loading
- Broken links (file not found) show a toast — no crash

**Dependency:** This feature is most useful alongside the folder sidebar (Feature 4 in this file). In single-file mode it works but is less discoverable.

---

## 6. Remote URL Preview

The app can fetch and render a markdown file from a remote URL.

**Rough scope**

- "File → Open URL…" accepts an `https://` URL pointing to a raw markdown file
- The app fetches the content, renders it, and displays it like a local file
- Remote images referenced in the document render (fetched over the network)
- The document is read-only — no file watching, no write-back
- A "Save Local Copy…" option saves the fetched content to disk
- Failed fetches (network error, non-markdown content type) show an error state

**Security note:** Remote content is untrusted. The HTML sanitizer (P1 Feature 8) must be on and non-overrideable for remote documents, even if users have disabled it for local files.

---

## 7. Editor Pane and Split View

The app gains an editor pane showing the raw markdown source alongside the rendered preview.

**Rough scope**

- View menu toggle: "Split View" shows editor (left) and preview (right) side by side
- Editor is a code editor component (CodeMirror 6 recommended) with markdown syntax highlighting
- As the user edits, the preview updates with the debounced re-render pipeline (P2 Feature 4)
- Scroll sync: scrolling either pane scrolls the other to the corresponding position
- The editor supports basic editing: typing, undo/redo, find-and-replace
- The file is saved to disk on `Cmd+S`; the file watcher suppresses re-render for self-triggered saves (same mechanism as P5 Feature 7)
- Viewer-only mode remains the default — split view is an opt-in

**Architectural note for earlier phases:** The file-write IPC path established in P5 Feature 7 (task write-back) is the foundation for editor save. Design it to handle arbitrary content writes, not just single-line replacements.

---

## 8. Paste Image from Clipboard

`Cmd+V` with an image on the clipboard saves the image to disk and inserts a markdown link at the cursor.

**Rough scope** (requires editor pane — Feature 7 — to be built first)

- Image on clipboard (PNG, JPG, GIF, WebP): saved to `./assets/<filename-timestamp>.png` relative to the open file
- `assets/` directory is created if it does not exist
- `![](./assets/image-2026-05-16.png)` is inserted at the cursor position in the editor
- Name collision: `-2`, `-3` suffixes added automatically
- Undo (`Cmd+Z`) reverts the text insertion; optionally prompts to delete the saved image file

**Architectural note:** The file-write IPC path from P5 Feature 7 handles the image file save. The editor cursor position is needed to determine where to insert the markdown link — this is a CodeMirror API call.

---

## 9. Diagram Inspector

Clicking a node inside a rendered Mermaid flowchart highlights the corresponding source definition line.

**Rough scope**

- Click on a flowchart node → the diagram's source definition is shown in a popover
- The specific node definition is highlighted within the source
- Available for flowchart and sequence diagram types initially; other types are out of scope
- Read-only — no editing in this feature (that is Feature 10)

**Architectural note:** Mermaid's SVG output includes `id` attributes on node elements that map back to the diagram source. The inspector parses this mapping to find the corresponding source line.

---

## 10. Mermaid Live-edit Popover

Clicking a diagram opens a mini editor for that diagram's source block. Saving writes the change back to the file.

**Rough scope**

- Click a diagram → a popover appears with the diagram source in an editable code field
- As the user edits, the preview inside the popover updates in real time
- "Save" writes the updated source back to the file, replacing the original ```` ```mermaid ```` block
- "Cancel" discards changes
- Uses the file-write IPC path from P5 Feature 7

**Architectural note:** The line-range write operation (replacing a multi-line block) is more complex than the single-line checkbox write. Plan for a `write-range` IPC variant when designing the P5 write path.

---

## 11. Presentation Mode

`---` horizontal rules act as slide boundaries, and the document is displayed as a full-screen slideshow.

**Rough scope**

- "View → Presentation Mode" (or `Cmd+Shift+P`) enters full-screen
- Each section between `---` dividers is one slide
- Arrow keys / space advance between slides
- Slide counter shown: "3 / 12"
- Pressing `Escape` exits presentation mode
- Mermaid diagrams, code blocks, and images render normally within slides
- Slide transitions are instant (no animation required initially)

---

## 12. Diff View

Two versions of a markdown file are shown side by side with visual differences highlighted.

**Rough scope**

- "File → Compare with…" opens a second file picker; the chosen file is compared to the current file
- Prose differences are highlighted line by line (added / removed / changed)
- Mermaid diagram blocks that differ show both the old and new diagram side by side
- Read-only — no editing in diff view
- A "Swap" button switches which file is considered "before" and which is "after"

**Architectural note for earlier phases:** No specific preparation is needed in earlier phases, but keeping the renderer stateless (re-renderable from any markdown string) makes building the diff view straightforward.
