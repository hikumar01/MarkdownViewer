# P5 Requirements — Extended Markdown Syntax

These features expand the set of markdown syntax the app understands beyond CommonMark and GFM. They are grouped into three plugin bundles to keep the renderer architecture manageable.

| # | Feature | Plugin Group |
|---|---|---|
| 1 | Footnotes (single-line) | R2 Block Extensions |
| 2 | Definition Lists | R2 Block Extensions |
| 3 | Abbreviations | R2 Block Extensions |
| 4 | Highlight / Mark | R1 Extended Inline |

| 6 | Subscript | R1 Extended Inline |

## Plugin Group R2 — Block-Level Extensions

Features 1–3 are shipped as a single toggleable plugin bundle. They are all non-GFM block constructs that do not conflict with each other or with GFM.

---

## 1. Footnotes (Single-line)

Footnote references in the body link to definitions at the bottom of the document.

**Syntax**

```markdown
This is a claim.[^1] Another claim.[^source]

[^1]: The source of the first claim.
[^source]: The source of the second claim.
```

**Acceptance Criteria**

- `[^label]` inline renders as a superscript link: `¹`
- Clicking the superscript scrolls to the footnote definition at the bottom of the document
- Footnote definitions render as a flat numbered list under a `---` divider and a "Footnotes" heading at the document end
- Each definition has a back-link (`↩`) that scrolls back to the inline reference
- Multiple references to the same label all link to the same definition
- Undefined labels (`[^missing]`) render as plain text — no crash
- Multi-paragraph footnotes are **out of scope** — only single-line definitions are supported
- Definition lines that are not referenced anywhere still render in the footnotes section

**Technical Notes**

`remark-gfm` includes GFM footnote support (enabled by default). If the project targets a stricter single-line-only behavior, the GFM footnote spec is sufficient — it does not require multi-paragraph support unless the author uses blank lines inside the definition.

No additional plugin is required beyond `remark-gfm`. The footnotes section is rendered automatically at the end of the document.

Ensure the footnote section `id` values are unique and do not collide with heading slugs (e.g., prefix footnote IDs with `fn-`).

**Dependencies:** remark pipeline (P0 Feature 1), anchor links (P1 Feature 5)

## 2. Definition Lists

Term-definition pairs render as semantic `<dl>/<dt>/<dd>` HTML.

**Syntax**

```markdown
Apple
: A fruit grown on trees.
: Also a technology company.

Orange
: A citrus fruit.
```

**Acceptance Criteria**

- Single term + single definition renders as `<dl><dt>Apple</dt><dd>A fruit...</dd></dl>`
- Single term + multiple definitions (multiple `: ` lines) renders multiple `<dd>` elements
- Multiple consecutive terms before a definition are each a `<dt>`
- Term (`<dt>`) is styled bold; definition (`<dd>`) is indented
- Regular paragraph text beginning with `: ` (colon-space) that is not part of a definition list is not affected
- Definition lists work at the top level and inside blockquotes

**Technical Notes**

Use `remark-definition-list`:

```js
import remarkDefinitionList from 'remark-definition-list'
processor.use(remarkDefinitionList)
```

This plugin detects the term-definition pattern in the AST and transforms it into `dl`/`dt`/`dd` nodes before rehype conversion.

**Dependencies:** remark pipeline (P0 Feature 1)

## 3. Abbreviations

A definition line causes every occurrence of an abbreviation in the document to show a hover tooltip.

**Syntax**

```markdown
Use GFM and HTML in your documents.

*[GFM]: GitHub Flavored Markdown
*[HTML]: HyperText Markup Language
```

**Acceptance Criteria**

- The definition line (`*[ABBR]: Definition`) is not visible in rendered output
- Every occurrence of the abbreviation text in the document body is wrapped in `<abbr title="Definition">ABBR</abbr>`
- Hovering over the abbreviation shows the native browser tooltip with the definition
- Matching is case-sensitive: `*[HTML]` matches `HTML` but not `html`
- Abbreviations inside fenced code blocks, inline code spans, and Mermaid source are excluded
- Multiple abbreviation definitions are all applied
- If the same abbreviation is defined multiple times, the last definition wins

**Technical Notes**

Use `remark-abbr`:

```js
import remarkAbbr from 'remark-abbr'
processor.use(remarkAbbr)
```

This plugin scans for `*[ABBR]: Definition` lines, removes them from the AST, and then walks all `text` nodes replacing matched substrings with `<abbr>` elements.

**Dependencies:** remark pipeline (P0 Feature 1)

## Plugin Group R1 — Extended Inline Syntax

Features 4–6 are shipped as a single toggleable plugin bundle. They all use single-character delimiters and share parsing logic via `remark-supersub` and a mark plugin.

**Parser conflict note:** Subscript (`~text~`) and GFM strikethrough (`~~text~~`) both use `~`. In the remark/micromark ecosystem, this is resolved structurally: micromark tokenizes `~~` as a delimiter run of length 2 and `~` as length 1 before any plugin runs. Plugin registration order does not affect this disambiguation. The edge case `~a~~b~~c~` produces `<sub>a<del>b</del>c</sub>` — correct behavior.

## 4. Highlight / Mark

Wrapping text in `==` renders it with a highlighted background.

**Syntax**

```markdown
This is ==highlighted text== in a sentence.
```

**Acceptance Criteria**

- `==text==` renders as `<mark>text</mark>`
- Light mode: yellow/amber highlight background (matching GitHub Discussions highlight color)
- Dark mode: adjusted highlight with sufficient contrast
- Works inside paragraphs, headings, list items, table cells, blockquotes
- Empty `====` does not crash — renders as plain `====`
- Does not conflict with ATX heading syntax (headings start at the beginning of a line with `#`)

**Technical Notes**

Use `remark-mark-and-unravel` or a custom micromark extension. Alternatively, implement a simple remark plugin using regex on text nodes — though AST-level is preferred for correctness:

```js
import remarkMark from 'remark-mark' // hypothetical — verify package name
processor.use(remarkMark)
```

If no maintained package exists, implement as a `micromark-extension` registering `==` as a delimiter construct.

**Dependencies:** remark pipeline (P0 Feature 1)

## 5. Superscript

`^text^` renders text as a superscript.

**Syntax**

```markdown
E = mc^2^
The 1^st^ of January
```

**Acceptance Criteria**

- `^text^` renders as `<sup>text</sup>`
- Works inline within paragraphs, headings, table cells
- Does not conflict with any other syntax — `^` is not used by CommonMark or GFM
- Nested inline (bold, italic) within superscript is supported: `^**bold super**^`
- Empty `^^` renders as plain `^^`

**Technical Notes**

Use `remark-supersub` which handles both superscript (`^`) and subscript (`~`) in a single plugin:

```js
import remarkSupersub from 'remark-supersub'
processor.use(remarkSupersub)
```

`remark-supersub` registers both constructs in a single micromark extension, ensuring they are processed in the same tokenizer pass.

**Dependencies:** remark pipeline (P0 Feature 1)

---

## 6. Subscript

`~text~` renders text as a subscript. Handled by the same plugin as superscript (Feature 5).

**Syntax**

```markdown
H~2~O is water.
CO~2~ is carbon dioxide.
```

**Acceptance Criteria**

- `~text~` renders as `<sub>text</sub>`
- `~~text~~` (GFM strikethrough) is not affected — rendered as `<del>text</del>` as normal
- `~a~~b~~c~` produces `<sub>a<del>b</del>c</sub>` — not garbled output
- Works inside paragraphs, table cells, list items
- Empty `~~` with a single-character gap (`~a~`) works correctly

**Technical Notes**

Handled by `remark-supersub` (same as Feature 5). No additional plugin required.

The conflict with strikethrough is resolved at the micromark level (delimiter run length). Verify with a test fixture:

```
Input:  ~~strike~~ and H~2~O and ~a~~b~~c~
Output: <del>strike</del> and H<sub>2</sub>O and <sub>a<del>b</del>c</sub>
```

**Dependencies:** remark pipeline (P0 Feature 1), superscript (Feature 5 in this file)

## 7. Task List Write-back

Clicking a rendered checkbox toggles its state in the source `.md` file on disk.

**Acceptance Criteria**

- Clicking an unchecked `[ ]` checkbox updates it to `[x]` in the rendered view immediately (optimistic update)
- The source file is written within **300ms** of the click
- The file watcher debounce is suppressed for self-triggered writes — the re-render that follows a write does not cause a visible reload loop
- If the file write fails (permissions error, file deleted): the checkbox reverts to its previous state and a toast is shown with the error
- Nested task lists (task list items inside a task list) work correctly
- Task lists inside blockquotes and callouts work correctly
- The file write replaces only the specific line containing the checkbox — no reformatting or rewriting of surrounding content

**IPC Path Design**

This feature establishes the file-write IPC path used by future features (e.g., P7 paste image). Design it generically.

The renderer annotates task list items with their source line numbers during the remark pipeline, then calls the Rust backend via `invoke()` on checkbox click:

```js
// remark plugin: annotate task list items with source line numbers
visit(tree, 'listItem', (node) => {
  if (node.checked !== null) {
    node.data = { hProperties: { 'data-line': node.position.start.line } }
  }
})
```

```typescript
import { invoke } from '@tauri-apps/api/core'

document.addEventListener('change', async (e) => {
  const checkbox = e.target as HTMLInputElement
  if (!checkbox.matches('input[type="checkbox"][data-line]')) return
  const lineNumber = parseInt(checkbox.dataset.line!, 10)
  const checked = checkbox.checked

  try {
    await invoke('toggle_task', { filePath: currentFilePath, lineNumber, checked })
    suppressNextReload() // prevent file watcher from triggering a visible reload
  } catch (err) {
    checkbox.checked = !checked // revert optimistic update
    showToast(`Write failed: ${err}`)
  }
})
```

```rust
#[tauri::command]
async fn toggle_task(file_path: String, line_number: usize, checked: bool) -> Result<(), String> {
    let content = fs::read_to_string(&file_path).map_err(|e| e.to_string())?;
    let mut lines: Vec<String> = content.lines().map(String::from).collect();
    if let Some(line) = lines.get_mut(line_number - 1) {
        if checked {
            *line = line.replacen("- [ ]", "- [x]", 1).replacen("* [ ]", "* [x]", 1);
        } else {
            *line = line.replacen("- [x]", "- [ ]", 1).replacen("* [x]", "* [ ]", 1);
        }
    }
    fs::write(&file_path, lines.join("\n")).map_err(|e| e.to_string())
}
```

Add `fs:read-files` and `fs:write-files` to `src-tauri/capabilities/default.json` (already required by P0 Feature 8).

**Dependencies:** remark pipeline (P0 Feature 1), auto-reload (P1 Feature 3)

## 8. Frontmatter Display

YAML or TOML frontmatter at the top of the document is parsed and shown in a collapsible metadata panel rather than rendered as a code block.

**Syntax**

```markdown
---
title: Architecture Overview
author: Engineering Team
date: 2026-05-16
tags: [architecture, backend]
---

# Architecture Overview
```

**Acceptance Criteria**

- YAML frontmatter (delimited by `---`) is detected and parsed
- TOML frontmatter (delimited by `+++`) is detected and parsed
- Parsed key-value pairs are displayed in a styled metadata panel at the top of the document, above the rendered content
- The panel is collapsible — default state (open/closed) is configurable in settings
- Frontmatter is **not** rendered as a prose code block — it is entirely consumed by this feature
- Unknown or malformed frontmatter (invalid YAML/TOML): the raw content is shown in a code block with a parse-error label
- Nested YAML objects render as indented key-value pairs (one level deep); deeply nested objects are shown as inline JSON

**Technical Notes**

Use `remark-frontmatter` to parse and remove frontmatter from the AST, then `remark-extract-frontmatter` (or a custom visitor) to extract the parsed data:

```js
import remarkFrontmatter from 'remark-frontmatter'
import remarkExtractFrontmatter from 'remark-extract-frontmatter'
import { parse as parseYaml } from 'yaml'

processor
  .use(remarkFrontmatter, ['yaml', 'toml'])
  .use(remarkExtractFrontmatter, { yaml: parseYaml, name: 'frontmatter' })
```

After processing, `file.data.frontmatter` contains the parsed object. The renderer injects a metadata panel element above the markdown content using this data.

**Dependencies:** remark pipeline (P0 Feature 1)

## 9. Mermaid Theme Switching

Users can choose the Mermaid diagram theme independently of the document light/dark mode.

**Options:** Default, Dark, Forest, Neutral, Base

**Acceptance Criteria**

- Mermaid theme is selectable in View menu → "Diagram Theme" submenu
- The selected theme is persisted in user settings and restored on relaunch
- Switching the theme re-renders all diagrams on the current page immediately — no reload required
- When the app is in light mode, the default Mermaid theme is `default`; in dark mode it is `dark`
- When a manual Mermaid theme is selected, it overrides the auto theme — persists across light/dark mode changes
- A "Follow App Theme" option resets to the automatic behavior

**Technical Notes**

```js
function setMermaidTheme(theme) {
  mermaid.initialize({ startOnLoad: false, theme })
  rerenderAllDiagrams()
}
```

`rerenderAllDiagrams()` re-processes all `.mermaid-source` elements in the DOM with the new theme. Because Mermaid caches rendered SVGs by ID, generate new IDs or clear the cache before re-rendering: `mermaid.mermaidAPI.reset()`.

**Dependencies:** Mermaid rendering (P0 Feature 2), light/dark mode (P1 Feature 1)

## Open Points

### OP-18 — Task Write-back Undo
Feature 7 specifies that a failed write reverts the checkbox, but does not address undo for a **successful** write the user immediately regrets. `Cmd+Z` behavior is currently undefined. Since the write modifies the source file, undo would need to re-toggle the checkbox and re-write the file. Decision needed:

- **Option A (recommended for v1):** No in-app undo — the change is intentional and immediate. Show a brief tooltip on the checkbox on first interaction: "Changes are saved to disk instantly." The user's external editor's undo handles this if needed.
- **Option B:** In-app undo via a write history stack — supports `Cmd+Z` to re-toggle and re-write. More complex; introduces state management for file mutations.

If Option A is chosen, document it explicitly in the feature AC so it is not treated as a bug during QA.

### OP-19 — Frontmatter Word Count Consistency
Feature 8 (Frontmatter Display) adds a visible metadata panel above the document. P3 Feature 3 (Status Bar) specifies that word count "excludes frontmatter." These two features need an explicit consistency decision:

- Even when the frontmatter panel is visible to the user, **word count must always exclude frontmatter key names and values** — the panel is metadata, not prose content.
- This should be an explicit acceptance criterion added to both P3 Feature 3 and P5 Feature 8 to prevent ambiguity during QA.

### OP-20 — App Packaging and Distribution
No requirement file covers how the app is built and shipped. This must be resolved before the first release:

**✅ Resolved by Tauri v2 framework decision.** `tauri build` handles packaging for all platforms.

- **Build command:** `tauri build` — produces platform-native bundles from a single command
- **macOS output:** `.app` bundle + `.dmg` disk image
- **Windows output:** `.msi` (WiX) + `.exe` (NSIS) — both generated by default
- **macOS code signing:** Set `APPLE_SIGNING_IDENTITY` env var with the Developer ID certificate name — Tauri signs automatically during `tauri build`
- **macOS notarization:** Set `APPLE_ID`, `APPLE_PASSWORD`, `APPLE_TEAM_ID` env vars — Tauri CLI calls `notarytool` automatically
- **Auto-update:** Add `tauri-plugin-updater` — configure a GitHub Releases endpoint as the update server:

```json
// tauri.conf.json
{
  "plugins": {
    "updater": {
      "endpoints": ["https://github.com/your-org/markview/releases/latest/download/latest.json"],
      "dialog": true,
      "pubkey": "YOUR_PUBLIC_KEY"
    }
  }
}
```

- **Distribution channel:** Direct download `.dmg`/`.exe` for v1; Mac App Store requires additional entitlement review — defer MAS to post-v1
- **CI integration:** Use the `tauri-apps/tauri-action` GitHub Action for automated builds, signing, and notarization on every release tag
