# P1 Requirements — Core Viewer Experience

These features are required for a coherent first public release. Without them the viewer is functional but feels incomplete or unpolished.

| # | Feature |
|---|---|
| 1 | Light / Dark Mode (OS Sync) |
| 2 | GitHub-style Typography |
| 3 | Auto-reload on File Save |
| 4 | Graceful Mermaid Fallback for Unknown Syntax |
| 5 | Clickable Anchor Links |
| 6 | File Type Association |
| 7 | Remember Window State |
| 8 | Raw HTML Passthrough with Sanitization |

## 1. Light / Dark Mode (OS Sync)

The app automatically follows the OS appearance setting and switches without restart.

**Acceptance Criteria**

- App starts in the mode matching the current OS setting
- When the user changes OS appearance (System Settings on macOS), the app switches immediately — no restart required
- All UI elements update on switch: prose text, backgrounds, code blocks, Mermaid diagrams, app chrome
- Mermaid diagrams re-render with the updated theme (`default` for light, `dark` for dark) on every OS appearance change
- Code block syntax highlighting switches between `github-light` and `github-dark` on theme change
- Transition between themes is instant — no flash of unstyled content

**Technical Notes**

Tauri provides a first-class theme API — no Rust code needed for the common case:

```typescript
// Frontend — src/theme.ts
import { getCurrentWebviewWindow } from '@tauri-apps/api/webviewWindow'

const appWindow = getCurrentWebviewWindow()

// Apply initial theme on load
const theme = await appWindow.theme()  // 'light' | 'dark' | null
applyTheme(theme ?? 'light')

// React to OS theme changes
await appWindow.onThemeChanged(({ payload: theme }) => {
  applyTheme(theme)
})

function applyTheme(theme: 'light' | 'dark') {
  document.documentElement.classList.toggle('dark', theme === 'dark')
  mermaid.initialize({ startOnLoad: false, theme: theme === 'dark' ? 'dark' : 'default' })
  rerenderAllDiagrams()
}
```

CSS approach — theme class on `<html>`:

```css
:root { --bg: #fff; --text: #1f2328; }
html.dark { --bg: #0d1117; --text: #e6edf3; }
```

No IPC bridge needed — `appWindow.theme()` and `onThemeChanged` are built into the Tauri JS API.

**Dependencies:** P0 complete

## 2. GitHub-style Typography

The rendered document uses the same typographic system as GitHub's markdown preview: font stack, sizing, spacing, and prose width.

**Acceptance Criteria**

- Prose maximum width: **768px**, horizontally centered
- Body font: system font stack (`-apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif`)
- Body font size: 16px, line height: 1.6
- Code font: `'SFMono-Regular', Consolas, 'Liberation Mono', Menlo, monospace`, size 85% of body
- Headings: H1 2em / H2 1.5em / H3 1.25em / H4 1em — all bold, with a bottom border on H1 and H2
- Blockquote: left border 4px, muted text color, indented
- Tables: full width, bordered cells, alternating row background
- Inline code: monospace, rounded background pill, no border
- Horizontal rule: 1px border, muted color
- Links: blue in light mode, lighter blue in dark mode; underline on hover
- Images: max-width 100%, height auto
- Task list checkboxes: aligned with list item text

**Technical Notes**

Ship a single `github-markdown.css` file modeled on GitHub's published stylesheet. Apply it scoped to the content container (e.g., `.markdown-body`) to avoid leaking into app chrome.

The stylesheet must define both light and dark variable sets:

```css
.markdown-body { --color-fg-default: #1f2328; --color-canvas-default: #ffffff; }
html.dark .markdown-body { --color-fg-default: #e6edf3; --color-canvas-default: #0d1117; }
```

**Dependencies:** P0 complete, Light/Dark mode (Feature 1 in this file)

## 3. Auto-reload on File Save

When the open file changes on disk, the rendered view updates automatically.

**Acceptance Criteria**

- File changes are detected within **500ms** of the file being written to disk
- The view re-renders the full document on change (incremental re-render is P4)
- Scroll position is **not** preserved on reload at this priority level — that is P2
- No manual refresh action is required from the user
- File watcher stops watching when the file is closed or replaced by a different file
- File watcher is restarted when a new file is opened
- Watching a file that is later deleted: show a "file not found" state — do not crash

**Technical Notes**

File watching runs in the Rust backend using the `notify` crate — native FSEvents on macOS, ReadDirectoryChangesW on Windows. More reliable and lower overhead than chokidar.

```rust
// src-tauri/src/commands.rs
use notify::{Watcher, RecursiveMode, recommended_watcher};

#[tauri::command]
async fn watch_file(path: String, window: WebviewWindow) -> Result<(), String> {
    let path_clone = path.clone();
    let mut watcher = recommended_watcher(move |res: notify::Result<notify::Event>| {
        match res {
            Ok(event) if event.kind.is_modify() || event.kind.is_create() => {
                window.emit("file-changed", &path_clone).unwrap();
            }
            Ok(event) if event.kind.is_remove() => {
                window.emit("file-deleted", &path_clone).unwrap();
            }
            _ => {}
        }
    }).map_err(|e| e.to_string())?;

    watcher.watch(Path::new(&path), RecursiveMode::NonRecursive)
        .map_err(|e| e.to_string())?;

    // Store watcher in app state to keep it alive
    Ok(())
}
```

```typescript
// Frontend — listen for file-changed event
import { listen } from '@tauri-apps/api/event'
import { invoke } from '@tauri-apps/api/core'

await listen('file-changed', () => reloadFile())
await listen('file-deleted', () => showFileDeletedState())
await invoke('watch_file', { path: filePath })
```

**Dependencies:** P0 complete, remark pipeline (P0 Feature 1)

## 4. Graceful Mermaid Fallback for Unknown Syntax

New or unrecognized Mermaid diagram types do not crash the document — they degrade gracefully.

**Acceptance Criteria**

- A diagram type not recognized by the installed Mermaid version shows an informational placeholder (not an error) with the raw source
- The placeholder message indicates the diagram type was not recognized
- All other diagrams and document content render normally
- When Mermaid is updated to support the previously unknown type, the diagram renders on the next reload without code changes

**Technical Notes**

This is an extension of P0 Feature 3 (Mermaid Parse Error Display). The distinction is in messaging:

- Parse error (P0): the diagram type is known but the syntax is invalid
- Unknown type (this feature): the first token of the diagram definition is not a recognized Mermaid diagram type

Detect unknown type by checking `err.message` for "No diagram type detected" or similar Mermaid error strings and rendering a softer placeholder.

**Dependencies:** Mermaid error display (P0 Feature 3)

## 5. Clickable Anchor Links

Headings generate anchor IDs, and links to those anchors scroll the document to the heading.

**Acceptance Criteria**

- Every heading (H1–H6) gets a slug ID: `## Installation` → `id="installation"`
- Slug generation matches GitHub's algorithm: lowercase, spaces to hyphens, non-alphanumeric characters removed, duplicate slugs get `-1`, `-2` suffixes
- Clicking a `[link](#installation)` in the document scrolls to the heading with that ID
- The URL hash updates in the webview when an anchor is clicked (for future back/forward support)
- Clicking an anchor that has no matching heading: no scroll, no error
- External links (`https://...`) open in the system default browser — not inside the app
- Relative file links (`./other.md`) are not yet supported — show as plain unclickable text at this priority

**Technical Notes**

Add a rehype plugin that slugifies heading text and sets the `id` attribute:

```js
import rehypeSlug from 'rehype-slug'
processor.use(rehypeSlug)
```

`rehype-slug` uses the same algorithm as GitHub. For anchor click handling, intercept `click` events on `<a>` elements in the renderer:

```js
document.addEventListener('click', (e) => {
  const a = e.target.closest('a')
  if (!a) return
  if (a.href.startsWith('#')) {
    e.preventDefault()
    document.getElementById(a.getAttribute('href').slice(1))?.scrollIntoView()
  } else if (a.href.startsWith('http')) {
    e.preventDefault()
    import('@tauri-apps/plugin-shell').then(({ open }) => open(a.href))
  }
})
```

**Dependencies:** remark pipeline (P0 Feature 1)

## 6. File Type Association

`.md` and `.markdown` files are registered with the OS so the app appears in "Open With" and can be set as the default viewer.

**Acceptance Criteria**

- The app declares `.md` and `.markdown` file type associations in its bundle
- macOS: app appears in "Open With" context menu in Finder for `.md` files
- macOS: user can set markview as the default via "Get Info → Open With → Change All"
- A file dragged onto the app's Dock icon opens it in the app
- A file passed as a CLI argument (`markview README.md`) opens it
- Opening a file via association while the app is already running replaces the current file in the existing window (single-tab model)
- On **first launch only**: the app shows a one-time prompt — "Set markview as the default .md viewer?" — with Accept and Skip options. This prompt does not appear again after either choice.

**Technical Notes**

Declare file associations in `tauri.conf.json` — Tauri generates the correct `Info.plist` entries for macOS and registry entries for Windows automatically:

```json
{
  "bundle": {
    "fileAssociations": [
      {
        "ext": ["md", "markdown"],
        "name": "Markdown File",
        "role": "Viewer",
        "mimeType": "text/markdown"
      }
    ]
  }
}
```

Handle file-open events using `tauri-plugin-deep-link` (covers both macOS open-file events and CLI arguments):

```rust
// src-tauri/src/main.rs
app.plugin(tauri_plugin_deep_link::init())?;

app.on_open_url(|urls| {
    // Called when a .md file is opened via Finder, Dock, or CLI
    for url in urls {
        let path = url.to_file_path().unwrap();
        app.emit("open-file", path.to_string_lossy().to_string()).unwrap();
    }
});
```

```typescript
// Frontend
import { onOpenUrl } from '@tauri-apps/plugin-deep-link'
await onOpenUrl((urls) => {
  const filePath = urls[0]?.replace('file://', '')
  if (filePath) openFile(filePath)
})

**Dependencies:** Single tab window model (P0 Feature 6)

## 7. Remember Window State

On relaunch, the app restores the previous window position, size, and last open file.

**Acceptance Criteria**

- Window bounds (x, y, width, height) are persisted when the window is closed
- On relaunch, window opens at the same position and size
- If saved bounds are fully off-screen (e.g., secondary monitor disconnected), reset to centered on the primary display
- Last open file path is persisted; on relaunch the file is reopened automatically
- If the last file no longer exists: show an empty state with a prompt to open a file — no error crash or toast
- Scroll position restoration is best-effort and not required at this priority level
- State is stored in the user data directory as JSON — not in the file itself and not in the app bundle

**Technical Notes**

Use two Tauri plugins — `tauri-plugin-window-state` for bounds (automatic) and `tauri-plugin-store` for last-file path:

```rust
// src-tauri/src/main.rs
// tauri-plugin-window-state saves and restores window bounds automatically
// — no manual resize/move listeners needed
tauri::Builder::default()
    .plugin(tauri_plugin_window_state::Builder::default().build())
    .plugin(tauri_plugin_store::Builder::default().build())
```

```typescript
// Frontend — persist and restore last open file
import { Store } from '@tauri-apps/plugin-store'

const store = await Store.load('settings.json')

// On file open
await store.set('lastFile', filePath)
await store.save()

// On app start — restore last file
const lastFile = await store.get<string>('lastFile')
if (lastFile) {
  const exists = await invoke<boolean>('file_exists', { path: lastFile })
  if (exists) openFile(lastFile)
  else showEmptyState()
}
```

`tauri-plugin-window-state` handles the off-screen guard automatically — it validates restored bounds against available displays before applying them.

**Dependencies:** Single tab window model (P0 Feature 6)

## 8. Raw HTML Passthrough with Sanitization

HTML tags embedded in markdown are rendered — `<details>`, `<kbd>`, `<br>`, `<sup>`, custom `<div>` blocks. A sanitizer strips dangerous elements.

**Acceptance Criteria**

- Block-level HTML renders: `<details>/<summary>` (collapsible), `<div>`, `<table>`, `<figure>`
- Inline HTML renders: `<kbd>`, `<br>`, `<mark>`, `<sup>`, `<sub>`, `<span>`
- `<details>/<summary>` collapsible sections are functional (native browser behavior)
- Sanitization is **always on** — the following are stripped regardless of setting:
  - `<script>` tags and their content
  - `on*` event handler attributes (`onerror`, `onclick`, `onload`, etc.)
  - `javascript:` href values
  - `<iframe>`, `<object>`, `<embed>`
- Elements not on the allowlist are stripped but their text content is preserved
- User can disable HTML passthrough entirely in settings (raw HTML is then escaped and shown as code) — default is on

**Allowlisted elements (non-exhaustive):**
`a`, `abbr`, `b`, `blockquote`, `br`, `caption`, `code`, `col`, `colgroup`, `del`, `details`, `div`, `dl`, `dt`, `dd`, `em`, `figure`, `figcaption`, `h1`–`h6`, `hr`, `i`, `img`, `ins`, `kbd`, `li`, `mark`, `ol`, `p`, `pre`, `q`, `s`, `samp`, `section`, `span`, `strong`, `sub`, `summary`, `sup`, `table`, `tbody`, `td`, `tfoot`, `th`, `thead`, `tr`, `ul`, `var`

**Technical Notes**

Use `rehype-sanitize` with a schema based on GitHub's allowlist:

```js
import rehypeSanitize, { defaultSchema } from 'rehype-sanitize'

const schema = {
  ...defaultSchema,
  tagNames: [...defaultSchema.tagNames, 'details', 'summary', 'kbd', 'mark'],
}

processor.use(rehypeRaw) // parse raw HTML nodes
processor.use(rehypeSanitize, schema)
```

`rehype-raw` must run before `rehype-sanitize` to parse the raw HTML nodes that `remark-rehype` passes through. Order matters: `remarkRehype({ allowDangerousHtml: true })` → `rehypeRaw` → `rehypeSanitize`.

**Dependencies:** remark pipeline (P0 Feature 1)

## Open Points

### OP-08 — Accessibility Requirements
The original feature proposals included accessibility but it was not assigned to any priority file. Accessibility should be treated as **P1** — a launch requirement for enterprise and government users:

- **Screen readers (VoiceOver on macOS, NVDA on Windows):** Rendered HTML must use semantic elements — headings (`<h1>`–`<h6>`), lists (`<ul>`, `<ol>`), landmarks (`<main>`, `<nav>`). All non-decorative images must have `alt` text. Mermaid diagrams need `role="img"` and a `<title>` element (see P4 Open Points OP-16).
- **Keyboard-only navigation:** All interactive elements — TOC entries, checkboxes (P5), anchor links, search results (P3), lightbox close (P4) — must be reachable via `Tab` and activatable via `Enter` or `Space`.
- **Reduced motion (`prefers-reduced-motion`):** Must be respected across all animated features — Mermaid diagram animations, lightbox fade-in (P4 Feature 3), diagram zoom/pan transitions, theme switch. All become instant when this preference is set.
- **High contrast mode (Windows):** The app must not break visually when Windows High Contrast Mode is active. Avoid hardcoded border colors that disappear in high contrast; use `currentColor` or system color keywords.
- **Minimum click/tap target size:** Interactive elements (checkboxes, TOC links, search prev/next buttons) must be at least 24×24 px.

### OP-09 — Link Edge Cases
Feature 5 covers `#anchor` and `https://` links but leaves gaps in link handling:

- **`mailto:` links** — open the system mail client via `open() from @tauri-apps/plugin-shell`. Do not open in the webview.
- **`tel:` links** — pass to `open() from @tauri-apps/plugin-shell` and let the OS decide; on desktop macOS this is typically a no-op unless Handoff/iPhone is paired.
- **Relative file links (`./other.md`)** — currently specified as "unclickable plain text." This will appear broken to authors who write relative cross-links. **Recommended:** render as a visually styled link; on click, show an informational toast — "Folder navigation is not yet supported" — rather than silently doing nothing.
- **Unknown protocols (`ftp://`, `ssh://`, custom schemes)** — pass to `open() from @tauri-apps/plugin-shell` and let the OS handle; log a warning if `openExternal` rejects the protocol.

### OP-10 — Code Block Filename / Title Annotation
Some markdown authors annotate code blocks with a filename or title:

~~~
```js title="example.js"
```
~~~

No requirement addresses what happens to this annotation. Decision needed:

- **Option A (recommended for v1):** Strip the annotation silently — treat the language tag as `js` and ignore `title=`. Shiki supports this via the `lang` extraction step.
- **Option B:** Parse the title and render it as a small tab or label above the code block.

This decision must be made before implementing P0 Feature 4 (Shiki integration), as the language extraction regex must handle the `title=` token correctly either way.

### OP-11 — File Reload Visual Indicator
P1 Feature 3 specifies the document reloads on file change and P2 Feature 4 debounces it, but neither specifies any feedback to the user that a reload occurred. Options:

- **Option A:** Silent reload — no indicator (current implicit spec)
- **Option B:** Subtle content-area flash — brief opacity pulse (150ms) on the `.markdown-body` element
- **Option C:** Transient "Reloaded" toast at the top of the window (dismisses after 2s)

**Recommended: Option B** — silent reloads can feel unreliable (user unsure if their save was picked up); a subtle pulse is informative without being distracting. Must respect `prefers-reduced-motion` (no animation if set).
