These features make the app significantly more useful for long documents and repeated use. They address navigation, file management, and document context.

## 1. Floating Table of Contents with Scroll-spy

A collapsible sidebar panel listing all headings in the document. The active heading is highlighted as the user scrolls.

**Acceptance Criteria**

- TOC panel lists all H1–H6 headings found in the document
- Heading levels are visually indented to show hierarchy (H2 under H1, H3 under H2, etc.)
- Clicking a TOC entry scrolls to the corresponding heading
- As the user scrolls the document, the TOC highlights the heading currently visible at the top of the viewport (scroll-spy)
- Only one heading is highlighted at a time
- TOC panel is toggled via View menu → "Table of Contents" and `Cmd+Shift+T`
- Panel visibility is persisted in user settings and restored on relaunch
- TOC updates immediately when the document reloads (file change)
- Documents with no headings show an empty panel with a "No headings found" message — no error
- TOC panel is overlaid on the document (floating), not pushing the document layout

**Technical Notes**

Extract headings from the rendered DOM after each render:

```js
function buildToc() {
  const headings = [...document.querySelectorAll('.markdown-body h1, h2, h3, h4, h5, h6')]
  return headings.map(h => ({
    id: h.id,
    text: h.textContent,
    level: parseInt(h.tagName[1])
  }))
}
```

Scroll-spy using `IntersectionObserver`:

```js
const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) setActiveTocEntry(entry.target.id)
  })
}, { rootMargin: '-10% 0px -85% 0px' })

headings.forEach(h => observer.observe(h))
```

The `rootMargin` shrinks the intersection zone to a horizontal band near the top of the viewport, so only the heading currently "at the top" is considered active.

**Dependencies:** Anchor links (P1 Feature 5), remark pipeline (P0 Feature 1)

## 2. In-document Search (Cmd+F)

A search bar allows the user to find and highlight text within the rendered document.

**Acceptance Criteria**

- `Cmd+F` opens the search bar — positioned at the top-right of the window, floating above content
- Typing in the search field highlights all matches in the document in real time (as the user types)
- Match count is displayed: "3 of 12 matches"
- `Enter` or the down arrow advances to the next match, scrolling it into view
- `Shift+Enter` or the up arrow goes to the previous match
- Current match is highlighted with a distinct accent color; other matches have a lighter highlight
- `Escape` or clicking outside the search bar closes it and clears highlights
- Search is case-insensitive by default
- Search matches inside code blocks, headings, table cells, and callouts
- No results: search bar indicates "No matches" — no error
- Mermaid diagram SVG content is excluded from search (it is not human-readable text)

**Technical Notes**

Use the browser's `window.find()` API or the `mark.js` library for highlighting. `mark.js` is preferred because it handles partial matches across DOM nodes cleanly:

```js
import Mark from 'mark.js'
const instance = new Mark(document.querySelector('.markdown-body'))

function search(query) {
  instance.unmark()
  if (!query) return
  instance.mark(query, {
    caseSensitive: false,
    accuracy: 'partially',
    done: (count) => updateMatchCount(count)
  })
}
```

For next/previous navigation, collect all `.mark` elements and scroll the active one into view.

Handle `Cmd+F` in the renderer with a `keydown` listener — no global shortcut needed since search only applies to the focused window. The input element should be auto-focused when the overlay opens.

**Dependencies:** P1 complete (document must be rendered)

## 3. Status Bar — Word Count and Reading Time

A fixed bar at the bottom of the window shows live document statistics.

**Acceptance Criteria**

- Word count reflects the body text of the document:
  - Excludes YAML/TOML frontmatter
  - Excludes content inside fenced code blocks (including Mermaid source)
  - Excludes HTML attribute values
  - Counts contractions as one word, hyphenated words as one word
- Reading time is calculated at **200 words per minute**, rounded up to the nearest minute
  - Display: "~4 min read" (minimum "~1 min read")
- Both values update within **200ms** of a document reload
- The status bar is a fixed-height bar at the bottom of the window — document content sits above it
- Document layout does not reflow when the status bar is toggled (it does not push content)
- Status bar is toggleable via View menu → "Status Bar" and persisted in settings

**Technical Notes**

Count words from the remark AST (before HTML conversion) for accuracy:

```js
function countWords(mdast) {
  let count = 0
  visit(mdast, 'text', (node) => { count += node.value.trim().split(/\s+/).length })
  // Skip code, inlineCode, html, and yaml nodes
  return count
}
```

Alternatively, extract `textContent` from the rendered DOM excluding `<pre>`, `<code>`, and `.mermaid-diagram` elements.

**Dependencies:** remark pipeline (P0 Feature 1), auto-reload (P1 Feature 3)

## 4. Status Bar — File Metadata

The status bar shows the open file's size and last modified timestamp.

**Acceptance Criteria**

- File size displayed in human-readable form: "12 KB", "1.4 MB"
- Last modified timestamp displayed in relative form: "Modified 2 minutes ago", "Modified yesterday"
- Both values refresh when the file changes on disk (i.e., after a reload event)
- If no file is open, both fields show a placeholder dash `—`
- Relative timestamp updates without requiring a file reload (a periodic tick every 60 seconds is acceptable)

**Technical Notes**

Use Node's `fs.stat()` in the main process:

```js
const { size, mtime } = await fs.promises.stat(filePath)
```

For relative time formatting, use a lightweight library (`date-fns/formatDistanceToNow`) or a simple manual implementation — avoid pulling in a large dependency just for this.

**Dependencies:** Auto-reload (P1 Feature 3), status bar (Feature 3 in this file)

## 5. Status Bar — Active Heading and Scroll Position

The status bar shows which section of the document is currently visible.

**Acceptance Criteria**

- Displays the text of the nearest heading above the current scroll position, prefixed with `§`: `§ Installation`
- If no heading is above the scroll position (user is at the top), shows nothing or "Top"
- If the document has no headings, shows the scroll percentage instead: "42%"
- Updates in real time as the user scrolls — debounced to once per 100ms to avoid jank
- Text is truncated with ellipsis if the heading is too long to fit the status bar

**Technical Notes**

Track scroll position using a `scroll` event listener with `requestAnimationFrame` throttling:

```js
let rafPending = false
document.addEventListener('scroll', () => {
  if (rafPending) return
  rafPending = true
  requestAnimationFrame(() => {
    rafPending = false
    updateActiveHeading()
  })
})

function updateActiveHeading() {
  const headings = [...document.querySelectorAll('.markdown-body :is(h1,h2,h3,h4,h5,h6)')]
  const active = headings.filter(h => h.getBoundingClientRect().top <= 20).at(-1)
  statusBar.setSection(active?.textContent ?? null)
}
```

**Dependencies:** Status bar (Feature 3 in this file)

## 6. Drag and Drop File

A markdown file dragged from Finder (or any file manager) onto the app window opens it.

**Acceptance Criteria**

- Dragging a `.md` or `.markdown` file onto the window surface opens it
- A visual drop target indicator (overlay or border highlight) appears while a compatible file is being dragged over the window
- Dragging an unsupported file type shows a rejection cursor and no action is taken on drop
- Dragging multiple files: only the first `.md` file is opened; others are ignored
- Drop replaces the current file — same behavior as Cmd+O

**Technical Notes**

Handle drag events in the renderer:

```js
document.addEventListener('dragover', (e) => {
  e.preventDefault()
  const hasMarkdown = [...e.dataTransfer.items].some(item =>
    item.kind === 'file' && (item.type === 'text/markdown' || item.getAsFile()?.name.match(/\.md$/i))
  )
  document.body.classList.toggle('drag-over', hasMarkdown)
})

document.addEventListener('drop', (e) => {
  e.preventDefault()
  document.body.classList.remove('drag-over')
  const file = [...e.dataTransfer.files].find(f => f.name.match(/\.(md|markdown)$/i))
  if (file) await invoke('open_file', { path: file.path })
})
```

In Tauri, `file.path` is available from the drop event because the WebView runs with the `fs` capability enabled. Verify that `tauri://drag-drop` events (if used) are allowed in `capabilities/default.json`.

**Dependencies:** Single tab window model (P0 Feature 6)

## 7. Recent Files List

The app maintains a list of recently opened files for quick access.

**Acceptance Criteria**

- File menu includes a "Recent Files" submenu listing the last **10** files opened
- Each entry shows the filename and the shortened path (e.g., `~/Documents/project/README.md`)
- Clicking an entry opens that file (replaces current file)
- If the file no longer exists at the stored path, the entry is shown grayed out; clicking it shows a toast: "File not found"
- The list is ordered most-recently-used first
- The current open file is not shown in the recent list while it is open
- "Clear Recent Files" menu item clears the list
- The list persists across app restarts

**Technical Notes**

```typescript
import { Store } from '@tauri-apps/plugin-store'

const store = await Store.load('settings.json')

async function addToRecent(filePath: string) {
  let recent = await store.get<string[]>('recentFiles') ?? []
  recent = [filePath, ...recent.filter(p => p !== filePath)].slice(0, 10)
  await store.set('recentFiles', recent)
  await store.save()
  rebuildRecentMenu(recent)
}
```

The in-app recent files list in `tauri-plugin-store` is the cross-platform implementation. It works identically on macOS and Windows and covers the full acceptance criteria.

> **Platform Enhancement (post-v1):** Native OS recent-documents integration can be added per platform:
> - **macOS:** Call `NSDocumentController.noteNewRecentDocumentURL()` via the `objc2` crate to add the file to the Dock's "Recent Items" menu
> - **Windows:** Add the file to the Windows Jump List via `ICustomDestinationList`
>
> Both are additive enhancements on top of the store-based list. See ADR-007.

**Dependencies:** Open file (P0 Feature 5), window state (P1 Feature 7)

## 8. Open in Default Editor (Cmd+E)

Opens the current file in the system's default text editor or a configured preferred editor.

**Acceptance Criteria**

- `Cmd+E` and "File → Open in Editor" open the current file in the system default app for `.md` files
- If no `.md` association exists, falls back to the system default text editor
- The app remains open and continues watching the file — the editor opens alongside markview
- If no file is currently open, the menu item and shortcut are disabled
- A "Preferred Editor" setting allows the user to specify an application path (e.g., `/Applications/Visual Studio Code.app`) that takes precedence over the system default

**Technical Notes**

```rust
// app/src/commands.rs
#[tauri::command]
async fn open_in_editor(path: String, preferred_editor: Option<String>) -> Result<(), String> {
    match preferred_editor {
        Some(editor) => {
            std::process::Command::new(&editor)
                .arg(&path)
                .spawn()
                .map_err(|e| e.to_string())?;
        }
        None => {
            // Opens with system default app for .md — respects "Open With" on macOS
            opener::open(&path).map_err(|e| e.to_string())?;
        }
    }
    Ok(())
}
```

```typescript
// Frontend
import { invoke } from '@tauri-apps/api/core'
import { Store } from '@tauri-apps/plugin-store'

const store = await Store.load('settings.json')
const preferredEditor = await store.get<string>('preferredEditor')
await invoke('open_in_editor', { path: currentFilePath, preferredEditor })
```

Uses the `opener` crate (cross-platform, respects OS file associations). No shell plugin needed for this command — direct Rust process spawn is cleaner.

**Dependencies:** Open file (P0 Feature 5), remember window state (P1 Feature 7)

## Open Points

### OP-14 — TOC Maximum Depth Setting
Feature 1 lists all H1–H6 headings in the TOC panel, but long technical documents with many subheadings produce a noisy, hard-to-scan TOC. Decision needed:

- Should there be a setting to limit TOC rendering depth (e.g., show only H1–H3)?
- **Recommended:** Add a depth control (range 1–6, default 3) to the Preferences panel (see P2 Open Points OP-13). The TOC panel itself could also offer a quick toggle (e.g., "Showing H1–H3 ▾") without requiring a trip to Preferences.

### OP-15 — Performance Budget
Individual timing targets exist throughout the requirements (500ms reload detection, 300ms write-back, 200ms status bar update) but no overall performance budget is defined. Suggested targets to validate before each major release:

| Metric | Target |
|---|---|
| Cold start — app launch to first file visible | < 2 seconds |
| Time-to-first-render — 50 KB file | < 500 ms |
| Time-to-first-render — 1 MB file | < 3 seconds (with loading indicator) |
| Mermaid diagram render — single diagram | < 1 second |
| Memory ceiling — resident memory | warn at 500 MB |
| Re-render after file save — 50 KB file | < 300 ms (debounce excluded) |

These targets should be captured as automated benchmarks (e.g., Playwright with Tauri's WebDriver integration via `tauri-driver`) and run on each release build.
